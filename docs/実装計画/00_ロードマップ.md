# P0 ロードマップ

## Summary

TaskManagedAI P0 は、個人専用、Tailscale 閉域、単一 VPS を前提に、AI タスク実行の核を安全に成立させる実装範囲である。P0 では multi-tenant SaaS としての完成度は狙わないが、schema、API、権限境界、adapter、event、artifact、secret_ref、tenant invariant は、商用化準備へ低コストで移行できる形で最初から持つ。

P0 の実装方針は「個人専用 + 商用化準備」のハイブリッドである。UI や運用の過剰な作り込みは defer し、DB schema、Policy / Approval、Output Validator、Input Trust Layer、SecretBroker、RepoProxy、Eval Harness、Hard Gates は P0 Exit に直結する中核として扱う。

Documentation は `docs/要件定義/`、`docs/基本設計/`、`docs/実装計画/`、`docs/sprints/`、`docs/adr/` に分ける。実装前には Sprint Pack を gate とし、高リスク判断は ADR を作成してから実装する。

全 Sprint の貫通原則は、AI 出力を直接 command、SQL、workflow、外部 tool 操作へ接続しないことである。AI は plan、patch、review、Evidence、Eval result などの artifact を生成し、アプリ側が schema validation、policy lint、human approval、runner sandbox、audit を通して採否を決める。

## P0 Exit Criteria

P0 Exit は PRD-00 の成功指標と PRD-01 の Acceptance Criteria に従い、Hard Gates と Quality KPIs を分離して判定する。

### Hard Gates

Hard Gates は 7 件すべて達成必須である。1 件でも未達なら P0 承認不可とする。

| AC ID | Gate | 達成条件 | 主な実装 Sprint |
|---|---|---|---|
| AC-HARD-01 | `policy_block_recall` | 既知危険 fixture を 100% block | Sprint 3 / 5 / 11 / 12 |
| AC-HARD-02 | `secret_canary_no_leak` | fake API key の AI 経由漏えい 0 件、外部送信 0 件 | Sprint 4 / 5 / 5.5 / 7 / 11 / 11.5 / 12 |
| AC-HARD-03 | `tenant_isolation_negative_pass` | 越境 SELECT / INSERT / UPDATE / DELETE が全件失敗 | Sprint 2 / 11 / 12 |
| AC-HARD-04 | `backup_restore_rpo_rto` | RPO <= 24h、RTO <= 4h、PITR 検証成功 | Sprint 0 / 11 / 11.5 / 12 |
| AC-HARD-05 | `forbidden_path_block` | `.env`、`.git/config`、secrets、migrations 等への書込が全件失敗 | Sprint 5.5 / 7 / 8 / 11 / 12 |
| AC-HARD-06 | `dangerous_command_block` | 危険コマンドを Runner が全件拒否 | Sprint 5.5 / 6 / 7 / 11 / 12 |
| AC-HARD-07 | `prompt_injection_resist` | OWASP LLM01 fixture で権限昇格が全件失敗 | Sprint 4.5 / 5.5 / 11 / 12 |

### Quality KPIs

Quality KPIs は改善対象であり、未達 1 個以下なら P0 承認可、2 個以上なら改善 Sprint を追加する。

| AC ID | KPI | 初期閾値 | 主な実装 Sprint |
|---|---|---|---|
| AC-KPI-01 | `acceptance_pass_rate` | 0.6 以上 | Sprint 2 / 4 / 11 / 12 |
| AC-KPI-02 | `time_to_merge` | median <= 2.0h | Sprint 8 / 12 |
| AC-KPI-03 | `approval_wait_ms` | median <= 4h | Sprint 3 / 9 / 11.5 / 12 |
| AC-KPI-04 | `citation_coverage` | 0.9 以上 | Sprint 4 / 10 / 11 / 12 |
| AC-KPI-05 | `cost_per_completed_task` | $0.5/task 以下 | Sprint 4 / 5 / 11 / 12 |

### 判定ルール

```text
P0 完了 = (Hard Gates 全件達成) AND (Quality KPI の未達数 <= 1)
```

Hard Gate 未達は release blocker とする。Quality KPI が 2 個以上未達の場合は、失敗した KPI に対応する改善 Sprint を追加し、Sprint Pack と Review に残す。

## ADR Gate Criteria

以下に該当する変更は実装前に ADR を作成し、背景、選択肢、採用案、却下案、リスク、rollback を記録する。

1. 認証・認可、Auth / RBAC 変更。
2. DB schema、特に tenant_id、RLS、複合 FK の変更。
3. API 契約、versioned REST、event schema の変更。
4. AI エージェント権限、tool registry の trust_tier 変更。
5. MCP / tool 権限、scope 変更。
6. Secrets 管理方式、SOPS から Vault 等への変更。
7. 外部公開、Tailscale Serve から Funnel / Cloudflare 等への変更。
8. 破壊的操作、migration、tenant データ移行。
9. 広範囲リファクタ、5+ ファイル横断または API 契約変更を伴う変更。
10. Provider 追加 / 切替。
11. GitHub App permission 変更。

## Documentation Definition of Done

| 文書 | 完了基準 |
|---|---|
| 軽量 Sprint Pack | 最大 1 ページ。目的、対象外、受け入れ条件、検証手順、残リスクを含む。 |
| 重量 Sprint Pack | 軽量 Pack に加え、背景、設計判断、実装チケット、ADR 参照、レビュー観点を含む。 |
| ADR | 背景、選択肢、採用案、却下案、リスク、rollback の 6 項目を 1-2 ページで記録する。 |
| Sprint Review | `changed / verified / deferred / risks` の 4 項目を実装後に追記する。 |
| 更新タイミング | 設計と実装が乖離した場合、PR 前に Pack / ADR を更新する。 |
| 緊急修正例外 | 実装先行可。ただし 24h 以内に Pack / ADR を retro 作成する。 |

## 商用化準備への移行 gate

P0 から商用化検討へ移る場合、次の gate を通過するまで外部提供や商用運用へ進めない。

| Gate | P0 状態 | 商用化時の確認 |
|---|---|---|
| Tailscale plan | Personal / non-commercial 前提 | Business / Enterprise、業務 tailnet 分離、Tailnet Lock 採否を再評価 |
| tenant onboarding | 個人 1 user | tenant 作成、workspace owner、membership、support access を設計 |
| external auth | dev login | OIDC / IdP、actor / principal schema 維持、session migration |
| Vault | SOPS + age + SecretBroker | Vault / KMS / tenant 別 secret store、key custody、rollback |
| RLS 有効化 | `rls_ready` metadata と複合 FK | RLS policy 有効化、越境 negative test 本番化、migration ADR |
