> 本計画は「現状監査 (実コード根拠) → 設計判断確定 (ユーザー Q&A 2 ラウンド) → ultracode workflow による phased 計画策定」の成果物。各フェーズ突入時に workflow で詳細計画を立て、Claude Code agent team 並列 + worktree 隔離 + 各ストリーム Codex Quality Loop で実装する。

## 1. 概要 (現状 → ゴール C)

現状は「記録」(ticket/run 作成 → append-only persist、MCP tool 群、蓄積基盤) は自律で動くが、「実行」が構造的に欠落している (実コード照合済)。

**5 大ブロッカー (実コード根拠)**:

1. production AgentRun を駆動する worker 不在。`run_create` は production run を作るが enqueue されず永久 queued (`api_bridge.py:705`)。唯一動く shadow driver も MockProviderAdapter ハードコード + `run_mode=='shadow'` のみ claim + MCP 非公開の三重ブロック。
2. real provider adapter (openai_responses/anthropic_messages/gemini) は存在するが driver 未 wiring (`agent_run_driver.py:305` MockProviderAdapter 固定)、CLIAgentAdapter は不在。
3. orchestrator は provider/validation/repair/shadow_completion の 4 step のみで policy_lint/diff_ready/approval/runner step が物理不在、runner_mutation_gateway は production caller ゼロ。
4. human kill switch primitive は実装済だが caller ゼロ + in-process dict で cross-process kill 不能。
5. SecretRefRepository は get/list のみで create/insert 物理不在、settings UI read-only、taskhub_admin skeleton。

本計画はこの「記録は動くが実行が動かない」状態から、**CLI サブスク実行 (claude -p / codex exec を headless subprocess spawn) を主軸**とし API provider を副次とする実行エンジンを ProviderAdapter 抽象上に構築して、ゴール C (L1→L4) へ積み上げる。最優先マイルストーンは **local Mac 起動** (Phase 0)。自律実行解禁の安全弁 **kill switch フル版** (Phase 1) を先に固め、production enqueue 本格解禁は全前提充足後 (Phase 6) に一本化する。

## 2. 指導原則 (guiding principles)

- **品質 > 速度** (絶対教訓 2026-05-13): CRITICAL invariant 直結変更は codex-quality-loop mode=code で R{N} clean (CRITICAL=0/HIGH≤2) まで磨く。
- **安全弁先行原則**: kill switch フル版 (ADR-00048 cross-process supervisor) と必要な Hard Gate SUT が揃うまで production 自律実行を解禁しない。latch は **driver atomic claim point + provider 呼出前 preflight** まで貫通させ、engage→課金/駆動継続の窓を残さない。
- **CLI サブスク実行を主軸・API key を副次**: CLIAgentAdapter を ProviderAdapter 抽象に追加し run 単位で CLI/API 選択 (既定 CLI)。
- **CLI サブスク認証 = host-ambient (ADR-00058 確定、self-rotating OAuth のため)**: claude/codex の credential は CLI 自身が所有・in-place refresh する OAuth token のため SecretBroker 管理は構造的に不可能。**host worker が `~/.claude` / `~/.codex` を直読** (SecretBroker 境界外、TaskManagedAI が raw に触れない、worker env/argv/DB/log/audit 非露出)。per-run 監査は `cli_invocation_started` / `cli_process_completed` event。**broker-managed は static API key のみ** (副次・Phase 2)、**in-process broker-mediated `provider.call`** (ProviderAdapter の in-process HTTP、subprocess を起動せず API key を env/argv/temp-file に出さず broker 内部で消費、provider response のみ返す)。**どの経路でも raw secret が子 process env に入らない**。CLI 認証 vs API token の境界は ADR-00058 で正式決定済。**【accepted HIGH risk + gate (ADR-00058 R16)】** host-ambient は CLI 配下 AI/tool が credential file (`~/.codex/auth.json` 等) を prompt-injection で直接 read し exfiltrate するリスクが残る。per-agent 専用最小 HOME (他 secret 非同居) + launcher tool sandbox で credential path の AI/tool read を deny + exfiltration negative test + canary で軽減するが完全には防げない → **accepted HIGH risk (local 単一 user)**。**confidential 以上 payload を扱う自律実行は Phase 2+ の追加封じ込め (専用 OS user / sandbox policy 強化) 完了まで gate** (Phase 2 CLIAgentAdapter は payload_data_class だけ見て confidential を CLI に流さない)。
- **CLI サブスクでも Provider Compliance を迂回しない**: CLIAgentAdapter.execute() 内で compliance_gate + provider_request_preflight を subprocess spawn 前に通す。consumer plan の training_use=yes なら effective=public 強制。
- **AI 出力直結禁止**: artifact → schema_validated → policy_linted → diff_ready → approval_required → waiting_approval → runner の pipeline を必ず通す。CLI stdout は untrusted_content、approval 後のみ trusted_instruction 昇格。
- **server-owned boundary 維持**: caller-supplied 経路 (allowed_data_class / payload_data_class / expected_request_fingerprint / approval hash / actor 申告 / runner policy_pass) を signature レベル物理削除。Defense-in-Depth 4 layer。
- **over-claim 警戒** (恒久教訓): frontmatter completed を地上真実とせず実コード grep + test 実行で配線状態を都度検証。code 存在 + test green ≠ 全 acceptance 達成 (SP-013/014/015/016/024 が completed でも実配線は本計画で初実施)。
- **ADR-first**: ADR Gate 11 種該当変更は実装着手前に proposed → accepted 昇格。
- **段階的解禁**: production enqueue 解禁を post-validation pipeline (Phase 3) + kill switch フル版 (Phase 1) + Hard Gate SUT 配線 (Phase 5) と明示ゲート (Phase 6)。それまで shadow-only enqueue 厳守。
- **既存資産 fold**: PR#350 shadow driver / SP-006 cli_artifact / 既存 real provider adapter / orchestrator step / state machine / Provider Compliance gate / SecretBroker atomic claim / active_registry_worker_gate / SP-0046 SSE / SP-001-5 host-portable を再利用。
- **local Mac first**: VPS/モバイル/外部連携/SOPS/Draft PR/Docker isolated runner は deferred。

## 6. 横断的制約 (cross-cutting、全フェーズ)

- **不変条件維持**: AgentRun16状態+blocked_reason3 / ContextSnapshot10列 (snapshot_kind ライフサイクルを Phase3 で完成) / Provider Compliance reason_code 13種 / SecretBroker atomic claim + OperationContext fingerprint / tenant-project boundary (全 table tenant_id NOT NULL + 複合 FK) / approval 4整合 + self-approval 禁止 / AI-output-boundary pipeline / cross-source-enum 5+整合。**enum 自体は増やさない**。
- **server-owned boundary**: caller-supplied 経路を signature レベル物理削除し server 側再計算。Defense-in-Depth 4 layer。
- **per-actor 監査**: 全 audit/AgentRunEvent payload に actor_id/run_id/trace_id/correlation_id binding。Phase 4 で複数 AI を actor 区別、api_bridge.py 13 箇所 DEFAULT actor 全 sweep。assert_no_raw_secret 必須。
- **raw secret 非保存**: DB/log/artifact/audit/ContextSnapshot/runner stdout/CLI subprocess env/argv に出さない。CLI サブスク credential は **host-ambient** (host worker が file 直読、TaskManagedAI が raw に触れない、SecretBroker 境界外、ADR-00058)。broker-managed (API key) は in-process `provider.call` (ProviderAdapter HTTP、subprocess なし、API key を env/argv/temp-file に出さず broker 内部で消費、provider response のみ返す)。secret revoke material は durable reconciliation (`status='revoked' AND material_purged_at IS NULL` を idempotent purge、ADR-00059)。material_state lifecycle (writing/present/purging/purged) で false-present を防ぎ token issue/redeem は `material_state='present'` 必須。**host-ambient credential-file の prompt-injection exfiltration は accepted HIGH risk (ADR-00058)**: per-agent 専用 HOME + tool sandbox deny + exfiltration negative test で軽減、**confidential 以上 payload の自律実行は Phase 2+ 追加封じ込め完了まで gate**。
- **ADR-first**: ADR Gate 11 種は実装着手前に proposed→accepted。CLI 認証境界(Phase0)/sealed 解除(Phase4)/production 解禁(Phase6) は user 確認。
- **品質 > 速度**: CRITICAL invariant 直結は codex-quality-loop mode=code R{N} findings_zero。Codex 並列起動禁止 + 3 連続失敗自動停止。
- **over-claim 警戒**: frontmatter completed を地上真実とせず実コード grep + test 実行で配線状態を都度検証。
- **worktree 隔離 + コンフリクト回避**: ファイル領域分割で stream 専有定義 (agent_run_driver.py=driver core stream を Phase2/3/6 逐次マージ、orchestrator.py を Phase3/4 直列化、state_machine.py の additive edge を Phase1/3/6 順次 merge、config/provider_compliance.toml と taskhub_admin.py は逐次)、cross-source-enum/event_type 変更は単独 PR、interface 先行合意。docker 残骸は起動側が down、実運用 volume 保護。
- **CI gate**: backend = mypy + ruff + pytest + ci-smoke backend-quality (postgres + RUN_DB_TESTS=1 フル DB suite)、frontend = typecheck/lint/Vitest/Playwright。1 ticket=1 PR=1 worktree branch、main 直 commit 禁止、Claude 起票+採否判定、user merge。

## 7. 実装進行戦略 (execution strategy)

Claude Code agent team で複数セッションを並列起動し、各フェーズの parallel_streams を **worktree 隔離** (`bash scripts/worktree_setup.sh` で pnpm install + uv sync + SOPS 復号、約 10 分。`.worktreeinclude` で個人設定 copy) で実装する。コンフリクト回避は:

1. **依存グラフに従いフェーズ順を厳守** (Phase 0/1 を先行単独 merge してから後続着手、Phase 4 は Phase 3 完成後に直列化)。
2. **ファイル領域分割** で各 stream の専有ファイルを事前定義 (§6 worktree 隔離参照)。
3. **cross-source-enum 5+整合変更は必ず単独 PR に閉じ**、state_machine.py を複数フェーズが additive edge で触る (Phase1 emergency/Phase3 production/Phase6 provider_incomplete→running) ため順次 merge する直列依存を明示。
4. **interface 先行合意** — adapter interface (ProviderAdapter Protocol、CLI subprocess の sync/async 呼出規約と long-running subprocess が DB transaction/claim lease を跨がない境界)・registry interface (DB-backed registry と active_registry_worker_gate の置換/共存)・lease 列 interface を着手前に確定。
5. **同一ファイルの並行編集回避** — config/provider_compliance.toml と scripts/taskhub_admin.py は逐次化。

各 stream は **Codex Quality Loop 相当の品質ゲート**をフェーズ標準として組込: CRITICAL invariant 直結変更 (driver/SUT/Policy Engine/SecretBroker/state machine/kill switch/CLI 認証境界) は codex-quality-loop mode=code (review→impl→adversarial 3 phase) で R{N} findings_zero (CRITICAL=0/HIGH≤2)、heavy plan/ADR は mode=plan (2 phase)、PR は codex_pr_full_review.sh で 3 endpoint baseline 内容確認。Codex は並列起動禁止、3 連続失敗で自動停止し AskUserQuestion。
