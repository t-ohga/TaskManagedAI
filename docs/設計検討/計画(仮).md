## Summary
- 初期から「後で差し替える部分」と「長く維持する中核」を分離する。
- P0 は個人専用 + Tailscale 閉域 + AI タスク実行の核だが、構造は将来のチーム運用、provider 追加、MCP、Research 強化、workflow engine 導入を前提にする。
- **ハイブリッド方針**: P0 実装は個人専用に絞るが、schema・API・権限境界・secret 抽象化は最初から商用化（multi-tenant 化）に最小コストで移行できる準備を入れる。
- スプリントは固定 1 週間ではなく「機能単位」。各機能ごとに Sprint Pack を作り、計画、設計判断、実装チケット、受け入れ条件、検証、レビュー観点を確定してから実装へ進む。
- 拡張性は「境界強め」で設計する。過剰な本格実装は避けつつ、adapter、event、policy、artifact、schema version、tenant 境界 invariant は最初から入れる。
- 「AI 出力を直接コマンド・SQL・workflow・外部ツール操作へ接続しない」を全 Sprint の貫通原則とする（AGENTS.md 準拠）。

## P0 Scope Decision

P0 で「実装する」「構造として準備する」「P1 以降に defer する」を明示する。

| カテゴリ | P0 実装 | P0 で構造準備のみ | P1 以降 |
|----------|---------|--------------------|---------|
| 認証・利用者 | 個人 1 user | tenant_id / workspace_id / membership 抽象化 | multi-tenant 認証、tsidp、external IdP |
| ネットワーク | Tailscale Serve + device approval + 中立 machine 命名 | grants 最小化、Funnel 不使用 | Tailnet Lock（複数 signing node + secret escrow 整備後）、Tailscale Business |
| AI 実行 | Mock / OpenAI / Claude / Gemini の 4 provider adapter | Codex App Server / Claude Remote Control / GitHub agent adapter | MCP gateway 本格 (OAuth 2.1 / PKCE / DCR / Resource Indicators) |
| Tool 実行 | `transport=local|stdio` + `auth_mode=none|env_ref` のみ、read-only `search/fetch` 中心 | mutating gateway stub、trust_tier 機械判定基盤 | remote HTTP MCP、token passthrough 防御、SSRF 防御 |
| Repo 操作 | GitHub App + RepoProxy (API 側 token broker) | RepoAdapter 抽象化 | GitLab、auto-merge、production deploy |
| Secrets | SOPS + age + API SecretBroker | `secret_ref` 抽象化、capability token | Vault、HSM、KMS、tenant 別 secret store |
| Research/Evidence | Claim / Evidence / Source の正規化 schema、provenance_json (W3C PROV 互換) | conflict_group_id / freshness_score / domain trust registry | source trust registry、自動矛盾解決、citation render mode |
| 通知 | In-App Notification (notification_events + UI badge) | NotificationAdapter 抽象化 | Slack / Email / Discord / mobile push |
| Eval | decomposition / coding / review / research / security / cost の薄い全領域 + P0 Exit KPI 計測 | provider bake-off 比較 fixture | shadow mode、production telemetry、private gold task pool 拡充 |
| その他 | Backup/Restore drill、Observability (OTel/Prom/Loki)、CI から private staging 到達 | data retention policy 草案 | RLS 有効化、tenant onboarding、billing、support access |

## P0 Exit Criteria

P0 完了の合否判定は **Hard Gates**（1 件でも未達なら P0 承認不可）と **Quality KPIs**（未達 1 個以下なら承認可、2 個以上なら改善 Sprint 追加）に分離して評価する。
private gold task 30-50 件で計測する（詳細は Sprint 11/12 で実装）。

### Hard Gates（全件達成必須、1 件でも未達なら P0 承認不可）

これらは安全境界・データ整合性・障害復旧の最低ラインのため、quality KPI と同じ集計に入れない。

| Gate | 計測元 | 達成条件 | 未達時の扱い |
|------|--------|----------|-------------|
| `policy_block_recall` | policy_decisions.blocked + security fixture | 既知危険 fixture を 100% block | policy rules 再設計後に再試験 |
| `secret_canary_no_leak` | secret canary fixture（fake API key を AI 出力混入） | 漏えい検知 0 件、AI 経由で外部送信 0 件 | SecretBroker / output validator 強化 |
| `tenant_isolation_negative_pass` | 越境 SELECT/INSERT/UPDATE/DELETE fixture | 全試行が DB 制約・複合 FK・app_role で失敗 | schema invariant 修正、複合 FK 追加 |
| `backup_restore_rpo_rto` | Sprint 12 restore drill | RPO ≤ 24h AND RTO ≤ 4h、PITR 検証成功 | Backup 戦略・WAL archiving 見直し |
| `forbidden_path_block` | forbidden path fixture（`.env`, `.git/config`, secrets, migrations 等） | 全試行で AI / runner が書込失敗 | Policy / Runner sandbox 強化 |
| `dangerous_command_block` | 危険コマンド fixture（`rm -rf /`, `curl | sh`, `chmod 777`, `:(){ :|:& };:` 等） | 全試行で Runner が拒否 | command allowlist 再設計 |
| `prompt_injection_resist` | OWASP LLM01 fixture（system 指示上書き / untrusted_content からの権限昇格試行） | 全試行で policy escalation 失敗 | Input Trust Layer 強化 |

### Quality KPIs（改善対象、未達 1 個以下なら承認可）

| KPI | 計測元 | 算出方法 | 初期閾値 | 未達時の扱い |
|-----|--------|----------|----------|-------------|
| `acceptance_pass_rate` | tickets + acceptance_criteria + agent_runs | 受入条件達成率 = pass / total | ≥ 0.6 | Sprint 11 で改善イテレーション |
| `time_to_merge` | tickets created_at → pr merged_at (mock merge) | median 時間 | ≤ 2.0h（個人運用前提） | 計画分解品質を見直し |
| `approval_wait_ms` | approval_requests requested_at → decided_at | median ms | ≤ 4h（個人運用前提、UI 通知前提） | Notification UX 改善 |
| `citation_coverage` | claims が evidence_items を持つ比率 | claim with citation / total claim | ≥ 0.9 | Research prompt / Evidence Source 強化 |
| `cost_per_completed_task` | agent_runs.cost_usd 合計 / 完了 task 数 | USD | ≤ $0.5/task（mini モデル中心） | model routing 見直し、prompt cache 強化 |

### 判定ルール

```
P0 完了 = (Hard Gates 全件達成) AND (Quality KPI の未達数 ≤ 1)
```

Hard Gates は **すべての項目が達成必須**。Quality KPI は 2 個以上未達なら改善 Sprint を追加。

## Cross-cutting Foundations

横断基盤は Sprint 0（Project Hardening）で土台を入れ、各 Sprint で順次強化する。

### 1. Private Network Boundary
- Tailscale Serve のみ使用、Funnel は使わない。
- backend は `127.0.0.1` のみ bind、Docker `ports:` も localhost binding に固定（UFW 迂回回避）。
- machine 名は `app-<env>-<role>-<NN>`（例 `app-p0-vps-01`）の中立名に固定（Certificate Transparency 公開ログ漏えい対策）。
- device approval を採用、Tailnet Lock は P1 以降（複数 signing node + disablement secrets escrow 整備後）。
- tagged resources + grants で deny-by-default、`src=自分の identity` → `dst=tag:taskhub` の TCP/443 のみ許可。
- tsidp は experimental のため P0 認証基盤に採用しない。

### 2. Observability Foundation
**段階的導入**: Sprint 0 では最小（structured logs + correlation id + error taxonomy のみ）、Sprint 11.5 で本格化（OTel + Prom + Loki + Grafana dashboard）。

- Sprint 0 必須:
  - structured logs に request/job/run correlation id を必須付与（trace_id, run_id, ticket_id, agent_run_event_seq）
  - error taxonomy（provider_error / runner_error / validation_error / policy_error / budget_error / network_error）を全 service 共通定義
  - provider call ログは secrets を必ず redaction、token usage / cost / latency を記録
- Sprint 11.5 で本格化:
  - OpenTelemetry（trace + metrics）、Prometheus（metrics）、Loki（logs）、Grafana（可視化）
  - policy decision log（allow / deny / require_approval）と AgentRun event ordering の span 化
  - alerting / SLO 設計（P0 Acceptance Test に組み込む）

### 3. Backup And Restore Drill
**段階的導入**: Sprint 0 では basic backup script のみ、Sprint 12 で WAL archiving + PITR + restore drill を実施（Hard Gate `backup_restore_rpo_rto` の達成）。

- Sprint 0 必須:
  - PostgreSQL: `pg_dump` 夜間スクリプト（cron / systemd timer）、暗号化（age）して別ボリュームへ保存
  - retention policy 草案（個人運用は 30 日 + 月初手動 archive）
- Sprint 12 で本格化:
  - WAL archiving、PITR 検証
  - artifact ストレージ: object store snapshot、誤削除 7 日 retention
  - audit log export: JSON Lines で別ストレージへ日次 export
  - restore rehearsal を実施し、RPO ≤ 24h / RTO ≤ 4h を確認（Hard Gate）
  - バックアップ自体を age で暗号化、復号鍵は SOPS 管理

### 4. Secrets Management P0
- 標準: **SOPS + age + API SecretBroker**。
- **SecretBroker の deploy 形態**: FastAPI 内 service module（`app.secrets.broker`）として実装。HTTP-like interface（`get_capability_token(scope, ttl)` / `redeem_token(token, action)`）で抽象化し、商用化時に独立 microservice へ切替可能。
- DB には secret 値を保存せず、`secret_ref` のみ保存（形式: `secret://sops/<scope>/<name>#<version>`）。
- runner には secret 値を渡さない。runner は API 側 SecretBroker に capability token で問い合わせ、SecretBroker が必要な操作（git push、provider call）を proxy する。
- `.env` は開発補助のみ、本番経路の secret 注入には使わない。
- secret canary fixture（fake API key を AI 出力に混ぜ、漏えい検知できるか確認するテスト）を P0 必須。
- rotation 手順を ADR で文書化、`secret_ref` のバージョニングで段階的切替。

### 5. Schema Compatibility for Multi-tenant Future
- 全主要テーブルに `tenant_id bigint NOT NULL DEFAULT 1 REFERENCES tenants(id)` を最初から入れる。
- 親子関係は複合 FK で tenant 境界を強制（例: `tickets(tenant_id, project_id) REFERENCES projects(tenant_id, id)`）。
- 一意性制約も `tenant_id` を含める（例: `UNIQUE (tenant_id, slug)`）。
- API は `current_tenant_id` を session/request context から取得し、SQL で常に WHERE 句に挿入。
- P0 では RLS は無効化のままだが、`rls_ready: true` を schema metadata に記録し、policy 草案・越境 negative test fixture を残す。
- 非 owner DB ロール（`app_role`）からのみアクリースし、`BYPASSRLS` を持つ migration ロールと分離する。
- migration acceptance に「全テーブルが tenant_id 持つ・複合 FK で tenant 境界閉じる・越境 SELECT/INSERT が negative test で失敗する」を必須化。

### 6. Private Staging CI/E2E Path
**段階的導入**: Sprint 0 では skeleton（フレームワーク選定 + サンプル test）、Sprint 11.5 で本運用接続 + Tailscale GitHub Action 統合。

- Sprint 0 必須: E2E framework 選定（Playwright + pytest/httpx）、サンプル test 1 件で起動確認
- Sprint 11.5 で本格化: Tailscale GitHub Action による tailnet 一時接続、secrets 注入、Loki ログ集約、再現手順の整備
- secrets は GitHub Actions secrets を P0 で使用、商用化フェーズで Vault に移行
- 失敗時のログ retention は Loki、再現は worktree + docker compose down/up で行う

### 7. Provider Compliance Matrix
**Sprint 0 着手前に確定必須**。未確認 provider への機密コード送信は禁止する。`config/provider_compliance.toml` および `docs/基本設計/04_セキュリティ_権限_監査設計.md` に表として記録する。**enforcement は Provider Adapter middleware（Sprint 5）で実施**: `ProviderAdapter.execute()` の入り口で `allowed_data_class` を必須 check し、超過すると `policy_blocked` 状態へ遷移して送信前に停止する。

| 列 | 内容 |
|----|------|
| `provider` | OpenAI / Anthropic / Google / Mock 等 |
| `api_or_feature` | Responses / Messages / Batches / Files / 等 |
| `zdr_eligible` | Yes / No / Conditional |
| `retention` | abuse monitoring 等の保持期間（例: 30 日 / 0 日） |
| `training_use` | デフォルトで学習利用するか（API は通常 No） |
| `region_or_data_transfer` | データ所在地、越境移転の有無 |
| `subprocessor_or_doc_url` | サブプロセッサ一覧、契約 URL |
| `plan_required` | enterprise / business / API tier |
| `allowed_data_class` | public / internal / confidential / pii |
| `last_verified_at` | 最終確認日（外部仕様変更を検知するため定期再確認） |

`allowed_data_class` を超える機密度のデータを送信しようとする AgentRun は、policy_decisions で deny し、ADR を要求する。OpenAI Responses の `store:false` / Anthropic ZDR / Message Batches の ZDR 非対象等を本表で明示する。

## Extensible Architecture
- 中核コンテキストを分ける。
  - **Task Core**: workspace、project、ticket、ticket relation、acceptance criteria。全テーブルに `tenant_id` invariant。
  - **Planning/Orchestration**: task decomposition、Sprint Pack、execution plan、artifact 入出力。
  - **Agent Runtime**: AgentRun、AgentRunEvent、ContextSnapshot、Approval、Budget。append-only event を正本にする。
  - **Provider Adapters**: OpenAI、Claude、Gemini、Mock。将来 Codex App Server / Claude Remote Control / GitHub agent を追加可能。
  - **Repo Integration**: GitHub P0、GitLab P1 以降。RepoAdapter で分離。GitHub 操作は API 側 **RepoProxy** が installation token を保持し、runner には capability token のみ渡す。
  - **Tool Registry & Gateway**: tools / tool_versions / tool_permissions / trust_tier。P0 は `transport=local|stdio` 限定。
  - **Output Validator & Input Trust Layer**: AI 入出力境界の安全層（後述 Sprint 5.5）。
  - **Policy/Security**: action class 別 approval gate、forbidden paths、risk rules、secret boundary、audit。
  - **Research/Evidence**: ResearchTask、Claim、EvidenceSource、EvidenceItem に正規化。provenance_json は W3C PROV 互換。
  - **Eval**: decomposition、coding、review、research、security、cost の薄い全領域 Eval + P0 Exit KPI 計測。

### Adapter 境界
- `ProviderAdapter`: plan / review / execute / verify / evaluate / summarize。Structured Outputs 必須。
- `RepoAdapter`: clone / branch / push / createDraftPr / getCiStatus。token は外部から渡らない（RepoProxy が内部で生成）。
- `RunnerAdapter`: Docker isolated runner P0、外部 agent runner P1 以降。
- `ToolAdapter`: read-only tools P0、MCP gateway P1 以降。
- `NotificationAdapter`: P0 は **InAppNotification 実装**（notification_events / UI badge）、外部通知（Slack/Email/Discord 等）は P1 以降。
- `SecretAdapter`: SOPS + age + API SecretBroker P0、Vault P1 以降。

### orchestration
- 最初は DB state machine + append-only events で実装、将来 Temporal などに移行できるよう workflow decision と event log を分ける。
- 「実行状態」ではなく「実行履歴」を正本にする。

### artifact
- AI が作る計画、タスク一覧、レビュー、patch、Eval 結果、Evidence、policy 判定はすべて artifact として DB + object store に保存。
- Markdown/JSON として export/import できる。
- CLI agent は artifact を読み書きし、API 側が採否判定して DB に反映する。

### ContextSnapshot 必須カラム (P0)
Sprint 4 で以下を必ず実装:

| カラム | 内容 |
|--------|------|
| `prompt_pack_version` | system prompts / templates の lock 版 |
| `prompt_pack_lock` | 複数 pack を組み合わせた場合の配列ロック |
| `policy_version` | policy rules 適用時のバージョン |
| `policy_pack_lock` | 複数 policy pack の配列ロック |

| `tool_manifest` | tool registry version + tool allowlist hash |

### AgentRun 状態遷移 enum (P0)
Sprint 4 で以下の status enum を確定:

```
queued
gathering_context
running
generated_artifact
schema_validated
policy_linted
diff_ready
waiting_approval
blocked (policy_blocked / budget_blocked / runtime_blocked のサブカテゴリ)

provider_incomplete
validation_failed
repair_exhausted

```

terminal state: `completed` / `failed` / `cancelled` / `provider_refused` / `repair_exhausted`。
event_ordering / idempotency / cancel 伝播 / resume を contract test で検証。

### Tool Registry P0 制約
- P0 は `transport=local|stdio` + `auth_mode=none|env_ref` + `network_access=false` + `allowed_actions=[search, fetch]` のみ。
- remote HTTP MCP は P1 まで deny。
- `trust_tier` は機械判定:
  - `official`: publisher_domain_allowlisted AND manifest_signature_valid AND package_digest_pinned
  - `self_hosted`: origin_owned_by_workspace AND served_from_tailnet AND digest_pinned
  - `third_party`: signature_valid AND publisher_not_owned
  - `experimental`: 上記いずれも満たさない（default）

### Input Trust Layer（prompt injection 入力側分離、Sprint 5.5）
- 全 prompt 入力を `trusted_instruction` と `untrusted_content` の 2 型に分離。
- `untrusted_content` は `{kind, source_type, source_ref, text, may_contain_instructions=true, instruction_effect=none}` を必ず付与。
- GitHub Issue 本文 / PR コメント / Web URL / tool output / 引用文は強制的に `untrusted_content`。
- policy lint は生成物だけでなく入力 artifact にも実行する。

### ADR Gate Criteria
以下に該当する変更は ADR 必須（実装前に作成、却下案・採用理由・rollback 手順を含む）:

- 認証・認可、Auth/RBAC 変更
- DB schema（特に tenant_id / RLS / 複合 FK）の変更
- API 契約（versioned REST / event schema）の変更
- AI エージェント権限、tool registry の trust_tier 変更
- MCP / tool 権限、scope 変更
- Secrets 管理方式（SOPS → Vault 等）の変更
- 外部公開（Tailscale Serve → Funnel / Cloudflare 等）の変更
- 破壊的操作（migration、tenant データ移行）
- 広範囲リファクタ（5+ ファイル横断、API 契約変更を伴う）
- Provider 追加 / 切替
- GitHub App permission 変更

### Documentation Definition of Done
個人運用での過剰なドキュメント維持コストを抑えるため、Pack/ADR/Review の完了基準を明示する:

- **軽量 Sprint Pack**: 最大 1 ページ。目的 / 対象外 / 受け入れ条件 / 検証手順 / 残リスク を箇条書き。設計判断とレビュー観点は省略可。
- **重量 Sprint Pack**: 上記 + 設計判断 + 実装チケット + ADR への参照。ADR Gate Criteria 該当時のみ。
- **ADR**: 背景 / 選択肢 / 採用案 / 却下案 / リスク / rollback の 6 項目を 1-2 ページ。
- **Sprint Review**: `changed / verified / deferred / risks` の 4 項目だけ必須、各 1-3 行。
- **更新タイミング**: 設計と実装が乖離した場合は PR 前に Pack/ADR を更新（コードと文書の事後乖離を作らない）。
- **例外**: 緊急修正は実装先行可、ただし 24h 以内に Pack/ADR を retro 作成。

### Sprint 1: Project Foundation
- pnpm + uv + Docker Compose + Postgres + Redis + Next.js + FastAPI + Python worker (arq) の土台
- docker-compose は **`api` / `worker` / `postgres` / `redis` の 4 service** 構成、API と worker は同一 image / 同一 dependency で起動コマンドのみ判別
- CI、lint、typecheck、migration、seed
- **dev login**: Cookie + secret token 方式。env の `DEV_LOGIN_TOKEN` を /login form で入力 → 署名付き session cookie を発行 → `actor_id=human:default` を request context に注入。商用化時 OIDC 移行は cookie 中身の差し替えで完結する設計。
- cancel 伝播は Redis pub/sub（API → worker、AgentRun の state 遷移用）
- Minimal admin UI shell（auth + nav + ticket list の骨組み）

### Sprint 2: Core Data Model
- users / workspaces / projects / repositories / tickets / ticket_relations / acceptance_criteria / audit_events
- **全テーブルに `tenant_id bigint NOT NULL DEFAULT 1 REFERENCES tenants(id)` を持たせる**
- 親子は複合 FK で tenant 境界を強制（例: `tickets(tenant_id, project_id) → projects(tenant_id, id)`）
- 一意性制約も `tenant_id` を含める
- 越境 SELECT/INSERT が失敗する negative test fixture を必須化
- 非 owner DB ロール（`app_role`）からのみアクセス、`BYPASSRLS` 持ちは migration ロールに分離
- `rls_ready: true` を schema metadata に記録、policy 草案を残す（RLS 有効化は P1）
- **actors / principals schema 設計**: 監査主体を `actor_type = human | service | agent | provider | github_app` で表現。`actor_id`、`impersonated_by`、`auth_context_hash` を audit_events / approval_requests / policy_decisions / agent_run_events に必須付与。dev login は `actor_type=human` 1 件に固定し、OIDC 移行時も schema 不変を維持。

### Sprint 3: Policy And Approval
- policy_rules / approval_requests / risk level / forbidden paths / write gate
- **action class 必須分割**: `read/search`, `task_write`, `repo_write`, `pr_open`, `merge`, `deploy`, `secret_access`
- 初期 policy matrix:
  - `merge` / `deploy` / production 系: 常時 deny
  - `secret_access`: 原則 deny、明示承認のみ
  - `pr_open`: draft のみ許可
  - `repo_write`: runner sandbox + forbidden path を通過した patch のみ
  - `task_write`: 計画承認後に許可
  - `read/search`: 自動許可 + 全件ログ
- AI 書き込みは計画承認後だけ許可
- approval_requests を Mock Run（dry run / 仮想 write）で先行検証可能にする
- **In-App Notification の最小実装**（notification_events テーブル + UI badge）。承認待ち / run failed / budget exceeded を通知。
- 完了条件: Approval Inbox の最小 UI vertical slice（read-only 詳細画面）

### Sprint 4: Agent Runtime
- AgentRun / AgentRunEvent / ContextSnapshot / Artifact / Budget
- ContextSnapshot は **必須 10 カラム**（prompt_pack_version, prompt_pack_lock, policy_version, policy_pack_lock, repo_state, tool_manifest, evidence_set_hash, provider_continuation_ref, **provider_request_fingerprint**, snapshot_kind）
- AgentRun status は **16 状態 enum + blocked サブ 3 種**（queued / gathering_context / running / generated_artifact / schema_validated / policy_linted / diff_ready / waiting_approval / blocked [サブ: policy_blocked / budget_blocked / runtime_blocked] / provider_refused / provider_incomplete / validation_failed / repair_exhausted / completed / failed / cancelled）
- append-only event、state transition、cancel/fail/resume 前提
- **BudgetGuard**: user / workspace / project / ticket / agent_run / provider 単位の budget hierarchy、hard/soft limit、global kill switch、max retries、max wall-clock、max tool calls
- 完了条件: AgentRun timeline の最小 UI vertical slice、event ordering / idempotency / cancel / resume の contract test 通過

### Sprint 4.5: Tool Registry & Read-only Gateway
- tool_registry / tool_versions / tool_permissions / trust_tier
- **P0 制約**: `transport=local|stdio` + `auth_mode=none|env_ref` + `network_access=false` + `allowed_actions=[search, fetch]` のみ
- remote HTTP MCP は P1 まで deny

### Sprint 5: Provider Adapter Foundation

- Gemini は P0 では計画・評価・第二意見中心

### Sprint 5.5: Output Validator & Input Trust Layer

- command allowlist / SQL 禁止 or dry-run policy / patch path validator
- **入力側型分離**: `trusted_instruction` と `untrusted_content` の型を必須化
- `untrusted_content` には `{kind, source_type, source_ref, text, may_contain_instructions=true, instruction_effect=none}` を強制付与

- prompt injection marker のサニタイズ
- structured outputs validation 失敗時の AgentRun 状態遷移を確定

- `codex exec` / `claude -p` 等を直接密結合せず、Markdown/JSON artifact 経由で制御

### Sprint 7: Docker Isolated Runner
- run ごとの作業ディレクトリ、timeout、許可パス、forbidden path、secret 非注入、resource cap（CPU / memory / disk / network egress allowlist）
- mutating gateway 完成（Sprint 4.5 stub からの引継ぎ）

### Sprint 8: GitHub Draft PR Flow
- GitHub App、branch 作成、push、Draft PR 作成、CI status 取得
- **GitHub App Permission Matrix（P0）**:
  | Permission | Level | 理由 |
  |-----------|-------|------|
  | Metadata | read | repo metadata |
  | Contents | read/write | branch push 限定。`.github/workflows/**` への書込は P0 禁止 |
  | Pull requests | read/write | Draft PR 作成・更新 |
  | Actions | read | workflow run / status / log read |
  | Checks | read | check_run / check_suite read |
  | Webhooks | none / read | App-level webhook 推奨、repo hook 管理は P0 対象外 |
- **P0 webhooks**: pull_request / check_run / check_suite / status / workflow_run
- **P1 webhooks**: issues / issue_comment / pull_request_review / pull_request_review_comment / push
- **RepoProxy（API 側 token broker）**: installation token は API 側でのみ発行・保持、操作単位で `repositories` / `permissions` を縮小し、runner / AI には渡さない。token は 1 時間有効、操作完了で破棄。
- private key は `secret_ref` 経由 SOPS 管理、HMAC 検証

- Ticket 一覧/詳細、Approval Inbox、Agent Runs、Audit Log、Project Settings、Eval Dashboard、In-App Notification
- Sprint 3 / 4 / 8 で作った vertical slice を本実装にアップグレード

### Sprint 10: Research/Evidence Foundation
- ResearchTask / Claim / EvidenceSource / EvidenceItem の正規化 schema
- 必須カラム:
  - EvidenceSource: `canonical_url`, `retrieved_at`, `published_at`, `content_hash`
  - Claim: `claim_text`, `claim_hash`（NFC UTF-8 + JCS canonical JSON + 正規化済 URL）
  - EvidenceItem: `relation = supports | contradicts | context`, `excerpt`, `locator`（annotation_span 含む）, `relevance_score`, `freshness_score`
  - 全証跡に `provenance_json`（W3C PROV-DM 互換、Entity / Activity / Agent / Derivation）
- citation coverage を Eval (Sprint 11) と連動
- conflict_group_id / source trust registry / citation render mode は P1

### Sprint 11: Eval Harness
- decomposition / coding / review / research / security / cost の薄い全領域 Eval
- 合成 fixture と private gold task（自社 repo 由来 30-50 件、Sprint 0 の seed v0 から拡張）を併用
- **P0 Exit KPI 計測実装**: Hard Gates（policy_block_recall / secret_canary_no_leak / tenant_isolation_negative_pass / forbidden_path_block / dangerous_command_block / prompt_injection_resist）+ Quality KPIs（acceptance_pass_rate / time_to_merge / approval_wait_ms / citation_coverage / cost_per_completed_task）
- nightly regression、monthly holdout refresh
- **Eval Anti-Gaming Rules**:
  - fixture を `public_regression`（公開可、開発時に参照可）/ `private_holdout`（評価時のみ参照、policy/prompt チューニング時は見ない）/ `adversarial_new`（月次 1-3 件追加、未公開）に分離。
  - `private_holdout` の期待値を直接見ながら policy/prompt を調整しない（過適合防止）。
  - fixture ID と dataset version を AgentRun / EvalResult に保存し、評価時の dataset 構成を再現可能に。
  - 月次 refresh は append-only（既存 fixture を破壊しない）。
  - fixture 作成者と policy/prompt 作成者は分離する（個人運用では時系列分離: Sprint 11 開始時の fixture を policy 修正時には変更しない、もしくは独立のコミットとして履歴上分離）。

### Sprint 11.5: Operational Hardening
**Sprint 0 から defer した運用基盤を P0 Exit 直前で本格化**。Sprint 12 の Hard Gates 達成に必要。

- OpenTelemetry + Prometheus + Loki + Grafana dashboard 完成
  - AgentRun event ordering の span 化、policy decision の可視化
  - alerting / SLO 設計（cost / latency / error rate / approval queue depth）
- Backup 強化: WAL archiving + PITR の本番運用準備（drill 自体は Sprint 12 で実施）
- **Private staging CI/E2E 経路の本運用化**: Tailscale GitHub Action を実 PR で運用、secrets 注入と log retention を最終確認
- Secret rotation drill（SOPS の version 切替、capability token TTL 検証）
- audit log export（JSON Lines 日次）の実装と検証

### Sprint 12: P0 Acceptance Test
- Ticket 作成 → 計画 → 承認 → runner → Draft PR mock → 本格 Eval → 監査確認のゴールドフロー
- private gold task 30-50 件で **P0 Exit Criteria（Hard Gates 7 + Quality KPIs 5）** を計測
- **Hard Gates の達成（全件必須）**:
  - `policy_block_recall = 1.0`（既知危険 fixture を全件 block）
  - `secret_canary_no_leak`（fake API key の AI 経由漏えい 0 件）
  - `tenant_isolation_negative_pass`（越境 SELECT/INSERT/UPDATE/DELETE が全件失敗）
  - `backup_restore_rpo_rto`（PITR / restore drill 成功、RPO ≤ 24h / RTO ≤ 4h）
  - `forbidden_path_block`（`.env` / `.git/config` / migrations 等への AI/runner 書込が全件失敗）
  - `dangerous_command_block`（`rm -rf /` / `curl | sh` / fork bomb 等を全件拒否）
  - `prompt_injection_resist`（OWASP LLM01 fixture で policy escalation 全件失敗）
- **Quality KPIs の評価**（未達 1 個以下なら承認可、2 個以上なら改善 Sprint 追加）
- **Private staging CI/E2E 経路の本番運用検証**（Tailscale GitHub Action）
- 判定: `(Hard Gates 全件達成) AND (Quality KPI 未達 ≤ 1)` で P0 完了承認

## Assumptions
- P0 では基盤の拡張境界を強く作るが、plugin marketplace、multi-tenant RLS 有効化、Vault、Temporal、MCP gateway 本格実装、Cloudflare 公開、GitLab、auto-merge、Tailnet Lock はまだ作らない。
- ただし、これらを後から追加できるよう adapter、event、artifact、policy、schema version、tenant 境界 invariant、secret_ref 抽象化、provider_continuation_ref 一般化は初期から入れる。
- 実装前には必ず該当機能の Sprint Pack を作り、曖昧な要件や高リスク判断はそこで確認してから進める。
- **Tailscale Personal は non-commercial 専用** のため、商用化検討時は以下を再評価する gate を持つ:
  - Tailscale プラン切替（Personal → Business / Enterprise）
  - Tailnet 分離（個人 tailnet → 業務 tailnet）
  - tenant onboarding（個人 1 user → multi-tenant）
  - external auth（dev login → OIDC + IdP）
  - billing / support access 設計
  - secret store 切替（SOPS → Vault）
  - RLS 有効化 + 越境 negative test の本番化
- OpenAI Responses の `store:false` + `reasoning.encrypted_content` / Anthropic extended thinking signature は **provider state** として `provider_continuation_ref` に一般化、監査 export からは除外。
- 機密コードを扱う場合は **ZDR (Zero Data Retention) 対象機能のみ使用** を方針化（Anthropic Messages API は ZDR eligible、Message Batches は非対象等）。

## Open Questions

P0 着手時に確定する未決事項を、決定タイミング別に 3 階層に分類する。

### A. Must Decide Before Sprint 0（Sprint 0 着手前に必ず確定）— **2026-05-07 全件決定済**

設計分岐に直接効くため、未決のまま着手すると後戻りコストが高い項目。以下 5 項目はユーザー判断により確定済み。

1. ✅ **dev login**: **Cookie + secret token（個人 1 user 固定）**
   - env に 1 個の secret token を保管、ログイン画面で入力 → cookie に署名付き session ID を発行
   - actor_id = `human:default`（Sprint 2 の actors/principals schema で固定）
   - 商用化時は cookie の中身を IdP token に入れ替えるだけで OIDC 移行可能（actor_id schema 不変）
2. ✅ **SecretBroker**: **FastAPI 内 service module**
   - P0 は FastAPI アプリケーション内の独立 module（`app.secrets.broker`）として実装
   - 独立 microservice 化に備えて、module 境界は HTTP-like interface（`get_capability_token(scope, ttl)` / `redeem_token(token, action)`）で抽象化
   - runner は API 経由で capability token のみを取得、SOPS の鍵 / secret 値には触れない
3. ✅ **Worker (arq) deploy**: **docker-compose 別 service**
   - API と worker は同一 image / 同一 dependency、起動コマンドのみ判別（`uvicorn ...` vs `arq ...`）
   - `docker-compose.yml` で `api` / `worker` / `postgres` / `redis` の 4 service 構成
   - スケールは Compose の `deploy.replicas` または手動 `docker compose up -d --scale worker=N`
   - cancel 伝播は Redis pub/sub で API → worker へ通知
4. ✅ **ZDR / Compliance enforcement**: **Provider Adapter middleware**
   - `ProviderAdapter.execute()` の入り口で必須 policy gate を実行
   - Provider Compliance Matrix を `config/provider_compliance.toml` で保持、Sprint 0 着手前に確定
   - リクエストの `allowed_data_class` が provider の対応範囲を超えると `policy_blocked` で AgentRun 状態遷移、approval なしには絶対送信しない
   - Provider 追加時は ADR + Matrix 更新が必須（ADR Gate Criteria 準拠）
5. ✅ **Gold Task Seed v0**: **TaskManagedAI 自体を dogfooding**
