## 10. Autonomy Level Matrix (R8 user 確定で追加、U-14)

(R10 番号再付番: 旧 §11 → §10、旧 §10 関連資料 → §11)

User 要求: **approval 不要で AI が自動実行できる範囲を 4 段階で設定切替可能にする**。

### 10.1 設計原則 (不変条件 #2 維持)

- **human-only approval decider invariant は維持**: approval を要する action では decider は依然 human only。
- **auto-allow path は別軸**: Policy Engine が `effect=allow` を返した action は `approval_requests` row を作らず、decider 欄なし。これは「approval を skip」であって「agent が decider に昇格」ではない (ADR-00009 §Tier 2 準拠)。
- **level 切替は ADR Gate Criteria #4 (AI エージェント権限)** に該当、**ADR retro は不可、accepted ADR + human approval before effect が必須** (R22 T-P2R1-001 反映、ADR Gate Criteria 11 種は break-glass 対象外)。default は最も安全な L0、user が project settings で切替は accepted ADR-00025 + accepted Sprint Pack + human approval event を全て満たした後でのみ有効化。
- **level に関わらず human approval 必須の action_class**: `secret_access` / `merge` / `deploy` / `provider_call` (L0〜L3 全 level で必須)。これは §2 #4 (SecretBroker raw secret 非保存) / §2 #5 (Provider Matrix) / §2 #17 (Realtime/Gemini direct 不可) の延長。
- **low-risk profile の機械判定**: 各 level で auto-allow される action_class でも、内部の low-risk profile 通過が必須。判定軸:
  - `payload_data_class <= internal` (confidential/pii は L 問わず approval 必須)
  - diff size <= N lines / file count <= M (level ごとに上限設定、ADR-00025 で固定)
  - forbidden path no-hit (`.env`、`.git/config`、secrets、migrations、`.github/workflows/**`)
  - dangerous command no-hit (Runner / CLI 経路の denylist)
  - `provider_request_preflight` PASS (canary、token pattern 検出なし)
  - `runner_mutation_gateway` 通過 (Sprint 7 後の本実装)
  - ContextSnapshot 10 列 PASS + `evidence_set_hash` 既定
- **kill-switch**: 各 level に対して BudgetGuard cap zero、Provider Matrix row blocked、Tool/MCP Gateway deny の 3 段 kill が揃う必要あり (T-029 mitigation)。

### 10.2 Level matrix (4 段階)

| Level | 名称 | auto-allow される action_class (low-risk profile 通過時) | human approval が必須の action_class | 想定用途 |
|---|---|---|---|---|
| **L0** | `strict` (default) | (なし、全 mutation で approval) | `task_write` / `repo_write` / `pr_open` / `secret_access` / `merge` / `deploy` / `provider_call` | 初回利用、新 provider 切替直後、高リスク環境、新 Sprint 着手直後 |

### 10.3 不変条件 (level 切替で破ってはならない)

L 設定値に関係なく以下は維持:

1. `secret_access` / `merge` / `deploy` / `provider_call` は **全 level で human approval 必須** (§2 #4 #5 #17)
2. `approval_requests.decided_by_actor_id` は **human actor のみ** (DB CHECK + service guard + Pydantic + pytest の 4 重防御)
3. auto-allow path で実行する action でも **AgentRunEvent + audit event** に `policy_profile` + `policy_version` + `auto_allow_reason` + `effective_action_class` + `applied_level` を必ず残す
4. low-risk profile の機械判定で **1 軸でも不合格** なら approval path に fall back (例: payload_data_class=confidential なら L1〜L3 でも task_write は approval)
5. level downgrade (L3→L0) はいつでも可能、upgrade (L0→L3) は **accepted ADR + accepted Sprint Pack + human approval event before effect** 必須 (R22 T-P2R1-001 + R24 T-P2R1-001-residual 反映、ADR retro は不可)
6. budget exceeded / global kill switch / Provider Matrix row blocked / Tool/MCP Gateway deny のいずれかが発火したら **level 設定を無視して即 `effect=deny` / AgentRun blocked** に切替 (R22 T-P2R1-003 反映: approval path 切替は不可、provider/tool 送信を fail-closed で完全停止)。復旧は **config change ADR + accepted Sprint Pack** 経由のみ、approval row 単独での再開は不可

### 12.1 設計原則 (不変条件 #1〜#18 厳守)

完全自立化しても以下は緩めない (§2 #1〜#18 全件適用):

1. **AI 出力直結禁止** (#1): 各 task の出力 (Codex review / patch / Sprint Pack draft / ADR draft) は artifact 化 → schema validation → policy lint → diff_ready → approval pipeline を通す。runtime 化 phase でも同じ。
2. **human-only approval decider** (#2): 各 high-risk decision point (Sprint Exit / ADR accepted 化 / autonomy upgrade / break-glass) は human approval 必須。`§10 autonomy level` で auto-allow される範囲は **low-risk profile 通過時のみ**。
3. **break-glass 不可** (#11 + §2 #11): ADR Gate Criteria 11 種は完全自立 workflow でも break-glass 対象外、緊急時でも先行不可。
4. **3 連続失敗で halt** (codex-multi-round-workflow §7): Codex 自動 round 中に 3 連続失敗で自動 halt + AskUserQuestion (sleep poll 禁止)。
5. **failure protection hook 維持**: `~/.claude/hooks/track-codex-failures.sh` の global counter は workflow 中も尊重、namespace 化しない (本 plan §10.4 server-owned-boundary 維持)。
6. **subagent 再帰禁止** (R16 F-P1R1-007 反映で正しい参照に修正: `.claude/reference/multi-agent-orchestration-draft.md:56-58` の max_depth / child status 制約、ADR-00014 §subagent 設計): workflow runtime は subagent 再帰 chain を作らない。メイン session → orchestrator → sub-skill の 3 段 chain まで。
7. **doc-only gate (§8) 維持**: 本 §12 は spec、実装着手は **SP-023 (新 Sprint Pack 候補、R16 F-P1R1-004 + R18 F-P1R1-004-residual 反映、SP-018 から変更)** accepted 後の別 run。(SP-018 は既存 Hermes memory sprint 予約済、衝突回避)

### 12.3 Task definition schema (`.claude/tasks/*.json`)

各 QL run / Sprint Pack BL は machine-readable task として記録:

```json
{
  "task_id": "QL-A-01",
  "parent_run": "QL-A",
  "title": "Sprint Pack canonical registry を docs/sprints/README.md に追加",
  "type": "doc_only",
  "write_scope": ["docs/sprints/README.md"],
  "depends_on": [],
  "blocks": ["QL-A-02", "QL-C-01"],
  "trigger_conditions": {
    "prerequisite_tasks_completed": [],
    "user_approval_required": false,
    "autonomy_level_min": "L0"
  },
  "codex_loops": [
    {"skill": "codex-review-loop", "max_rounds": 12, "clean_criteria": "critical_zero"},
    {"skill": "codex-adversarial-loop", "max_rounds": 12, "clean_criteria": "critical_zero"}
  ],
  "verification": {
    "acceptance_spec": ["registry 全 BL-* が §3.5.4 Pack inventory の actual_file に解決"],
    "no_test_file_changes_in_this_run": true,
    "no_code_changes_in_this_run": true
  },
  "rollback_condition": "doc-only revert (PR revert)",
  "invariant_trace": ["#11", "#16"],
  "estimated_codex_rounds": "3-5",
  "human_gates": ["Sprint Pack review approval (if heavy Pack)"]
}
```

Storage:
- **Immutable task definition** (`.claude/tasks/{QL,sprint10,sprint11,sprint11_5,sprint12,phase5}/<task_id>.json`、gitignored): 上記 schema、task graph 定義、人間レビュー対象。**authorization-affecting fields** (`trigger_conditions.user_approval_required`, `autonomy_level_min`, `human_gates`, `verification.acceptance_spec`) は **caller-supplied 不可 / server-resolved only** (R22 T-P2R1-004 反映、§2 #14 server-owned-boundary)。task file は signed artifact として **human-reviewed commit + hash 検証** が必須、orchestrator が runtime に load する前に signature 検証で reject 可能。
- **Runtime state ledger** (`.claude/audit/task_runtime/<task_id>.events.jsonl`、gitignored、orchestrator が append のみ、R22 T-P2R1-005 反映): **JSONL append-only event log + monotonic sequence + previous_hash chain**。各 event は `{ts, seq, prev_hash, event_type, sanitized_payload}` 形式。**current status は materialized view として再計算**、`<task_id>.state.json` 等の status 上書き型 file は禁止
- **状態遷移**: workflow task status enum は **`workflow_task_status`** (新 enum、R22 T-P2R1-010 反映、AgentRun.status とは物理分離、5+ source manifest 別 entry [DB CHECK + Pydantic + Python Literal + pytest EXPECTED + frontend])。値: `pending` / `in_progress` / `completed` / `blocked` / `failed`、命名衝突回避のため Pydantic / SQLAlchemy で **明示 namespace** (`workflow_task_status` 等)
- **責務分離**: task definition は immutable (PR で human review)、runtime state は append-only event source。orchestrator は event 追記のみ、status update は禁止。conflict 時は file lock + atomic rename + sequence/hash chain 整合性検証で integrity 保証
- **Audit ledger schema 制約** (R22 T-P2R1-009 反映): event payload に **raw token / raw secret / private key / raw OperationContext / unredacted provider request body** を含めない。`last_error` は raw text 不可、`sanitized_error_ref` (artifact id へのポインタ、artifact 自体に redaction_status + canary scan PASS が記録) のみ。`assert_no_raw_secret` 相当の canary scan を ledger write 直前に必須実行

### 12.4 Orchestration loop (Claude main session 半自動)

各 session 開始時に runtime ハーネスが以下を実行:

```
1. .claude/tasks/ から task graph load
2. 各 task の status check (pending / in_progress / completed / blocked / failed)
3. dependency graph 解析 → 次に着手すべき task を選出
4. trigger_conditions evaluate (prerequisite + user_approval + autonomy_level)
5. user_approval_required=true なら AskUserQuestion で確認
6. codex_loops 実行 (Skill chain、3 連続失敗で halt)
7. verification.acceptance_spec check
8. artifact / Sprint Pack / ADR update (Layer 3)
9. task status を completed に
10. 次 task へ (graph 探索、並列禁止)
```

**halt 条件** (auto-stop):
- 3 連続 Codex 失敗
- Hard Gate fixture failure
- 不変条件 violation 検出
- user_approval_required=true で human 不在
- session context 80% に達する (compact recommend)

### 12.5 Failure recovery + rollback + audit ledger (R22 T-P2R1-006/008/009/017 反映で強化)

各 failure 種別に **recovery (即時停止) + rollback_condition + rollback_action + post_rollback_verification** の 4 要素を明示:

| failure 種別 | 検知 | recovery | rollback_condition | rollback_action | post_rollback_verification |
|---|---|---|---|---|---|
| Codex 3 連続失敗 | `~/.claude/hooks/track-codex-failures.sh` global counter ≥3 | auto-halt + AskUserQuestion (継続 / 中止 / 別 task へ skip)。**counter reset は human approval event + reason + previous counter hash 必須** (R22 T-P2R1-006)。**counter file 削除 / missing は PASS ではなく BLOCK** | 当該 task の write 操作が完了済かつ partial state | event ledger に `rollback_initiated` append、対応する artifact / migration / Pack diff を revert | post-revert で task state を `failed` (上書きではなく event append)、Hard Gate fixture / acceptance spec で revert 後検証 |
| Hard Gate fixture failure | task verification step | 該当 task status=`failed` (event append) | fixture failure 内容で Sprint scope 内に rollback 必要なら | Sprint Pack `## Review` に rollback action 記録、artifact / migration revert | private_holdout / public_regression 再実行で PASS 確認 |
| 不変条件 violation | cross-source-enum-audit / postgres-boundary-audit / provider-compliance-audit | task halt (event append、status 上書きなし)、reject 判定、Sprint Pack `## Review` 記録 | violation が code/schema/migration に到達 | git revert + DB migration down (該当時) | violation 検出 audit を再実行で PASS、5+ source manifest 全件 exact-set check |
| ADR Gate Criteria 該当検出 (未起票) | adr-audit skill | task halt、新 ADR proposed 起票を user 提案 | proposed 起票で済めば revert 不要 | accepted 化が必要な変更は accept されるまで実装 revert | accepted ADR ref が Sprint Pack `## Review` で参照可能 |
| session context 80% | session token budget check | task 完了後に compact recommend、memory に handoff entry 書出 | (rollback 不要) | (該当なし) | 次 session で memory entry から再開可能 |
| **human 不在 (user_approval_required=true)** (R16 F-P1R1-005 + R22 T-P2R1-008 反映) | trigger_conditions.user_approval_required=true で session に human input なし | task status=`blocked` (event append)、reason=`waiting_human`、**auto-skip 不可**。**再開条件: `{approval_response_id, user_message_id, approval_target_artifact_hash, timestamp}` 全件 ledger 記録必須**、session presence / restart は resume 条件にしない | (rollback 不要、halt のみ) | (該当なし) | resume 時に上記 4 fields の ledger event 存在を必須検証、欠落時は再 blocked |

audit ledger format (`.claude/audit/sprint_progress.jsonl`、R22 T-P2R1-009 反映で redaction 制約強化):

```jsonl
{"ts": "2026-05-15T11:00:00Z", "seq": 1, "prev_hash": "0000...", "task_id": "QL-A-01", "event": "started", "actor_id": "<DB actor row id>", "actor_type": "human|service|agent", "session_id": "xxx", "redaction_status": "none_required", "canary_scan": "PASS"}
{"ts": "2026-05-15T11:15:00Z", "seq": 2, "prev_hash": "<sha256 of seq=1>", "task_id": "QL-A-01", "event": "codex_round_completed", "round": 1, "skill": "codex-review-loop", "findings_count": 5, "adopted": 5, "verdict": "needs_revision", "redaction_status": "applied", "canary_scan": "PASS", "sanitized_error_ref": null}

```

**Forbidden in audit ledger** (R22 T-P2R1-009): raw token / raw secret / private key / raw OperationContext / unredacted provider request body / Codex stdout/stderr raw / SecretBroker denial raw / runner stdout raw。これらは **artifact id 経由 (`sanitized_error_ref`)** で参照、artifact 自体に redaction_status + canary scan + payload_data_class metadata。

**actor_id 制約** (R22 T-P2R1-007): `actor_id` は **DB `actors` table row id 必須**、自由文字列 (e.g., `"claude-main-session"`) は invalid。`actor_type` は `human` / `service` / `agent` / `provider` / `github_app` の 5 enum 固定 (§2 #2 と整合)。

### 12.6 Sprint Exit gate (Layer 4 強化)

```

   - AC-HARD-01〜07 を private_holdout / public_regression / adversarial_new で全件実行
   - 1 件でも fail なら Sprint Exit BLOCK
2. Quality KPIs 5 集計
   - 未達 2 個以上で改善 Sprint を追加 (P0 完了条件)
3. open finding zero check (R22 T-P2R1-011 反映で valid defer schema 強化)
   - Sprint Pack 関連 task の review_results に **open + invalid_defer + unresolved_blocker** が全て 0
   - **invalid_defer 定義**: non-INFO finding を `defer` にしたが `{owner, impact, resume_condition, blocked_by, verification, target_hash}` のいずれかが欠落しているもの。欠落 1 件でも open 扱い
4. harness incident zero check (R22 T-P2R1-006 反映で tamper-evident 化)
   - `~/.claude/local/codex-failure-count.*` の counter が **tamper-evident** (counter file 削除 / missing / reset event なしの 0 = BLOCK)。reset event は human approval + reason + previous counter hash を含む event ledger row 必須
5. Sprint Pack `## Review` 章 生成
   - changed / verified / deferred / risks を audit ledger から自動集約
6. acceptance report 生成
   - Hard Gate fixture ID + dataset version + verdict 一覧
7. **human approval** (Layer 0 へ、R22 T-P2R1-007 反映で DB binding 強化)
   - 上記 1-6 PASS でも、Sprint Exit verdict は human が最終確定 (auto-pass 不可)
   - **Approval service row 必須**: `approval_requests` に該当 Sprint Exit target の row 作成 + `decided_by_actor_id` が **DB `actors.actor_type='human'` row** であることを DB CHECK + service guard + Pydantic + pytest の 4 重防御で enforce
   - **Target hash binding**: approval target は `{sprint_id, acceptance_report_hash, hard_gate_results_hash, kpi_aggregate_hash, audit_ledger_tail_hash}` で構成、approval row の `artifact_hash` に記録
   - **audit ledger には approval_id 参照のみ**、approval target の raw payload は含めない
```
