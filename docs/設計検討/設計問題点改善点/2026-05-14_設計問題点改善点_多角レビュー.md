# TaskManagedAI 現行設計の問題点・改善点 多角レビュー

作成日: 2026-05-14  
対象: `/Users/tohga/repo/TaskManagedAI` の要件定義、基本設計、ADR、Sprint Pack、設計検討文書、関連する一部実装境界  
目的: 現在の設計を、機能面、UX、AI 自律ワークフロー、CLI/ターミナル操作性、複数プロジェクト運用、設計 traceability の観点から評価し、次に直すべき問題と改善案を明確にする。

## 総合判定

TaskManagedAI の設計は、AI 出力を直接実行せず、Artifact、Policy、Approval、Audit、SecretBroker、Provider Compliance、Runner 境界に分ける安全設計がかなり強いです。AI-native な開発タスク管理ツールとして重要な「証拠、判断、承認、実行ログ、コスト、レビュー結果を残す」という方向性も一貫しています。

一方で、現時点の弱点は明確です。設計の中心が安全境界と実行統制に寄っており、日常的に使うタスク管理ツールとしての操作面、Deep Research から Ticket/PR へつながる E2E 受け入れ、ターミナル操作の context 解決、複数プロジェクト運用、完全自律性に向けた runtime gate の接続がまだ閉じていません。

結論として、現行設計は「安全な AI 実行基盤の設計」としては強いが、「毎日使える AI-native タスク管理プロダクト」としては、Task Operating Layer、Inbox/Attention、CLI context resolver、Research-to-Task lineage、autonomy policy profile を先に補う必要があります。

## 評価した設計の強み

1. **安全境界の思想が明確**
   - AI 出力を直接コマンド、SQL、workflow、外部 tool 操作へ接続しない方針が明文化されている。
   - Provider Adapter、Repo Adapter、Runner Adapter、Tool Gateway、SecretBroker、Approval を分ける設計は、将来の provider/agent 追加にも耐えやすい。

2. **監査可能性を中核に置いている**
   - AgentRun event、artifact、ContextSnapshot、approval、budget、provider fingerprint、audit log を残す発想は AI-native 開発管理として妥当。
   - Hard Gates/KPI を P0 受け入れに置いている点も、単なる UI プロダクトではなく運用可能性を重視している。

3. **将来拡張の境界が早めに切られている**
   - Multi-agent、provider compliance、tenant/project boundary、SecretBroker、runner sandbox、UI/CLI parity は、P0 ですべて作らないとしても設計上の逃げ道がある。

4. **P0 を個人 dogfooding、Tailscale 閉域、GitHub-only に絞っている**
   - 初期導入の現実性は高い。外部公開、multi-tenant SaaS、Slack/Discord 通知、full local LLM を後回しにしている判断は妥当。

## 問題点と改善案

### 1. 機能面、プロダクト面

#### DI-02: P0 Acceptance が Deep Research を通らない

現状の Product Vision は「Deep Research から実装 PR まで」を中核にしています。しかし P0 完了フローは `Ticket -> AI plan -> 承認 -> runner -> Draft PR -> Eval -> Audit` であり、ResearchTask、Claim、Evidence、citation coverage、generated ticket の流れが必須検収になっていません。

このままだと、通常の AI 実装管理ツールとして P0 完了できてしまい、TaskManagedAI の差別化要件である「調査から実装までの証拠付き管理」が検収されません。

改善案:

- SP-012 の P0 Acceptance に `ResearchTask 作成 -> Claim/Evidence 保存 -> citation_coverage 計測 -> generated ticket 作成 -> approved plan -> Draft PR` を must_ship として追加する。
- `citation_coverage` と `generated ticket 全件が根拠 Evidence を持つこと` を Hard Gate か feature-level acceptance に入れる。
- Research なしの Ticket-to-PR flow は smoke test とし、P0 の代表 gold flow にはしない。

#### DI-03: P0 後半の Sprint Pack が閉じていない

ロードマップやバックログでは RepoProxy/Draft PR、Research/Evidence、Eval Harness、Operational Hardening が P0 中核として参照されています。一方で、該当 Sprint Pack が存在しない、または名称がずれており、SP-012 が前提にする Pack も閉じていません。

改善案:

- `SP-008_github_draft_pr`、`SP-010_research_evidence`、`SP-011_eval_harness`、`SP-0115_operational_hardening` を作成する。
- もし P0 から外すなら、ロードマップ、P0 backlog、SP-012 acceptance から一貫して外す。
- `docs/sprints/README.md` に canonical sprint id registry を追加し、`sprint_pack_ref` と実ファイル名の drift を検出する。

#### DI-06: Ticket model が日常タスク管理として薄い

現在の Ticket model は status、priority、risk、issue_type、description、Acceptance Criteria を持ちます。AI 実行の最小単位としては成立しますが、日常的なタスク管理として必要な owner、reviewer、approver、label、component、due date、estimate、next_action、blocked_by、source/intake、branch/PR linkage、AI allowed actions、evidence links が弱いです。

改善案:

- P0 必須、P0 構造準備、P1 の三段階で Task Operating Layer を切る。
- P0 では物理カラムを増やしすぎない場合でも、view model として `next_action`、`owner_actor`、`blocked_by`、`due_at`、`aging`、`approval_policy`、`ai_allowed_actions`、`evidence_refs` を持たせる。
- `ticket_assignments`、`ticket_labels`、`ticket_estimates`、`ticket_ai_policy`、`ticket_source_events` を P0.1 以降の正規化候補として明記する。
- Acceptance Criteria は `DoD / test / docs / rollback / evidence` の構造化を検討する。

#### DI-07: Evidence-to-Task / Decision の正規化が不足

ResearchTask は findings、open questions、decision、generated tickets を扱う想定ですが、現行設計では `generated_ticket_ids jsonb` や `decision text` に寄っています。JSON 配列だけでは、どの証拠、どの判断、どの claim から、どの ticket や acceptance criteria が生まれたかを DB 制約で追いにくくなります。

改善案:

- `research_decisions` を追加し、decision、reasoning summary、accepted/rejected/deferred、decider、evidence_set_hash を保存する。
- `research_generated_tickets` を追加し、ResearchTask/Decision と Ticket の lineage を FK で結ぶ。
- `evidence_to_acceptance_criteria` を追加し、AC がどの evidence/claim に依存するかを追えるようにする。
- SP-010 の acceptance に「生成 Ticket 全件が根拠 Evidence を持つ」を入れる。

#### DI-10: review result が first-class artifact になっていない

レビュー結果は、AI-native workflow の採否、再レビュー、KPI に直結します。しかし現行の artifact kind は `report` に寄せる余地があり、review verdict、finding severity、target artifact/diff hash、required revision、再実行要否が構造化されていません。

改善案:

- P0 で `review` artifact kind か `review_results` schema を明示する。
- `verdict`、`findings_count`、`severity`、`target_artifact_id`、`target_diff_hash`、`required_revision`、`resolved_by_artifact_id` を持たせる。
- Reviewer Agent、人間レビュー、Eval review を同じ read model で扱う。

#### DI-11: cost ledger の再計算性が弱い

AgentRun には cost/tokens があり、provider usage event も想定されています。ただし、provider 単価改定、retry、Research run と Code run の按分、mock/real cost、ticket 単位集計を再計算できるかが曖昧です。

改善案:

- table 追加を P0 で避ける場合でも、usage event payload schema に `pricing_version`、`billable_usd`、`cost_source`、`ticket_id`、`provider_request_fingerprint` を必須化する。
- `cost_per_completed_task` の計算式を SP-011/SP-012 に fixture として持たせる。

### 2. UX/UI

#### DI-08: 日次運用の入口が弱い

SP-009 は Board、Ticket Detail、Approval Inbox、AI Runs の 4 面を定義しています。これは構成として妥当ですが、ユーザーが朝最初に見るべき「何を承認し、何を直し、何が詰まっているか」の control plane が弱いです。

改善案:

- Board を単なるカンバンではなく `Today / Inbox control plane` として定義する。
- 最低限、承認待ち、失敗 run、blocked、invalidated approval、高リスク ticket、SLA 超過、次アクションを優先順位付きで出す。
- 「読む通知」ではなく「処理する attention queue」として設計する。

#### DI-12: Approval UX が approve/reject に寄りすぎている

P0 Plan Mode は approve/reject に絞る方針です。安全上はシンプルですが、現実の承認判断には action class、risk、affected resource、diff summary、policy reason、validator state、evidence links、estimated cost、rollback note、stale state が必要です。

改善案:

- Approval card を decision packet として再定義する。
- 必須表示は `action_class`、`risk_level`、`affected resource`、`diff summary`、`policy reason`、`validator state`、`evidence links`、`estimated cost`、`rollback note`、`stale state`。
- Reject は理由または修正要求を残して再計画へ戻せるようにする。
- P0 で plan inline edit を入れないとしても、`request revision` は必要です。

#### DI-13: Execution Log が監査向けに重い

tools、citations、diffs、eval、policy、audit を同じ timeline に並べる設計は監査に強いです。一方、日常利用では「なぜ止まったか」「次に何をすればよいか」が raw event に埋もれます。

改善案:

- Timeline を 2 層に分ける。
- 上段は `要約`、`判断理由`、`ブロッカー`、`次アクション`、`影響範囲`。
- 下段は展開式の raw event、diff、citation、policy detail、audit payload。
- AI work の読みやすさは、ログ量ではなく「次の判断に必要な情報の圧縮」で評価する。

#### DI-14: 通知が queue になっていない

P0 通知は approval pending、run failed、budget exceeded、policy blocked、pr opened などが想定されていますが、severity、due、dedupe、target route、required action、actioned、snooze、escalation が弱いです。

改善案:

- `notifications` を in-app triage queue として拡張する。
- 最低限 `severity`、`due_at`、`dedupe_key`、`target_route`、`required_action`、`actioned_at`、`snoozed_until` を検討する。
- 外部通知は P1 defer でよいが、P0 in-app は実務用 queue にする。

#### DI-15: P0 で価値が見える dashboard が欠ける可能性

PRD と P0 Acceptance は KPI を重視しています。一方、SP-009 では業務価値 dashboard が defer になりがちです。フル dashboard は不要でも、dogfooding では「AI が役に立っているか」を毎日見られる必要があります。

改善案:

- SP-009 must_ship に簡易 KPI strip を追加する。
- 表示対象は `approval_wait_ms`、`acceptance_pass_rate`、`cost_per_completed_task`、`failed/blocked runs` 程度でよい。
- drill-down、分析、長期 trend は P1 defer でよい。

### 3. 完全自律性を高める AI ワークフロー

#### DI-05: 現時点で mutating autonomous work は BLOCK

設計上の思想は自律化へ向いていますが、現時点で provider call、secret access、runner/tool、multi-agent orchestration を AI が自律的に mutating 実行するには、runtime gate の接続が未完成です。

主な未接続リスク:

- `PolicyDecisionService.evaluate()` が runtime の provider/secret/repo/runner/tool 経路で必須 gate になっていない。
- Approval decision で `actor_type = human` が service/DB の中心境界として強制されていない。
- Provider compliance の `evaluate` と `enforce` 経路が分かれ、監査 payload が欠落しやすい。
- BudgetGuard が provider call 前の hard stop ではなく、事後記録寄りになる可能性がある。
- SecretBroker の approval 要件が action class policy と完全には一致していない。
- Runner sandbox / mutation gateway が完成するまでは、AI による command/file mutation/MCP mutation を許せない。

改善案:

- provider、secret、repo、runner、tool の外部送信または mutating action はすべて `PolicyDecisionService` を必須にする。
- missing rule、unknown resource、missing policy version は deny-by-default。
- ApprovalDecisionService は decider actor を lock し、human-only を service と negative test で強制する。
- Provider path は audited gateway に一本化し、allow/deny/preflight deny/provider send 直前の event を必ず残す。
- provider call 前に estimated max tokens/cost/wall-clock と global kill switch を評価する pre-spend gate を追加する。
- Runner/MCP mutation gateway 完成まで、AI による mutating command は設計上 BLOCK と明記する。

#### DI-09: 自律性レベルが定義されていない

「完全自律性」を高めたい一方で、このプロジェクトでは human approval、policy gate、SecretBroker、runner sandbox を中核にしています。そのため、全自動か手動かではなく、どの種類の作業をどの条件で自律化するかを明示する必要があります。

推奨する autonomy level:

| Level | 意味 | P0/P0.1 での扱い |
|---|---|---|
| L0 | read-only 調査だけ | 常時許可可能 |
| L1 | plan/artifact 作成まで自動、実行は人間承認 | P0 基本 |
| L2 | 低リスク ticket の修正案作成と dry-run まで自動 | P0 後半 |
| L3 | 低リスク allowlist 内の mutating run を policy/budget/review gate 付きで自動 | P0.1 候補 |
| L4 | orchestrator が複数 agent を割り当て、失敗時に自己修復する | P1 候補 |
| L5 | merge/deploy/secret/high-risk を完全自動 | 採用しない。常に human approval |

改善案:

- `autonomy_policy_profiles` を設計に追加する。
- profile は `allowed_action_class`、`required_review_artifacts`、`max_iterations`、`max_cost`、`max_wall_clock`、`allowed_project_scope`、`abort/escalation` を持つ。
- UI/CLI では run 開始時に autonomy level と stop condition を明示する。

#### DI-16: no-progress/loop/cost escalation の利用者体験が不足

ADR には lease、heartbeat、max_turns、kill-switch が出ていますが、ユーザーから見て「なぜ止まったか」「いつ止めるか」「コストがどこまで増えたか」が一貫した UX になっていません。

改善案:

- Run view と CLI に `pause`、`resume`、`cancel`、`kill`、`lower autonomy` を出す。
- `no_progress`、`budget_near_limit`、`approval_waiting`、`blocked_external` を attention queue に入れる。
- long-running run は `progress heartbeat` と `last useful artifact` を表示する。

#### DI-17: memory と project auto-discovery が自律性の鍵だが後ろ倒し

P1 memory は妥当ですが、複数プロジェクトで継続的に AI workflow を動かすには、project auto-discovery、repo mapping、context snapshot、local memory の境界が早めに必要です。

改善案:

- P0.1 で `project auto-discovery` の最小仕様を決める。
- memory は ContextSnapshot を置き換えず、supplement として扱う。
- project/repo boundary をまたぐ memory retrieval は deny-by-default にする。

### 4. CLI / ターミナル操作性

#### DI-04: 13 capability matrix が実体化されていない

ADR-00015 と SP-016 は UI/CLI parity と 13 capability を掲げていますが、13 項目の正本表がありません。どの UI action、API endpoint、CLI command、action class、audit event、parity assertion が対応するか不明です。

改善案:

- ADR-00015 に 13 行の正本表を追加する。
- 列は `capability_id`、`UI flow`、`API endpoint`、`CLI command`、`required flags`、`action_class`、`allowed_actions`、`project/repo scope`、`audit event`、`parity assertion`、`disabled/deferred behavior`。
- SP-016 の parity test はこの表を入力にして生成または検証する。

#### DI-18: SP-006 は user CLI ではなく CLI agent artifact bridge

SP-006 は AI agent の artifact orchestration であり、ユーザーが terminal から TaskManagedAI を操作する CLI ではありません。user CLI は SP-016 で P0.1 寄りです。このままだと dogfooding 初期に terminal 操作が弱くなります。

改善案:

- P0 に `tm-lite` または最小 user CLI を追加する。
- 最小範囲は `ticket list/show`、`run show`、`approval list`、`approval approve/reject`、`context show`。
- full parity、message、export、provider、sprint 操作は P0.1 のままでよい。

#### DI-19: ContextResolver が未設計

terminal では cwd、git remote、profile、env、明示 flag が食い違うことがあります。`tm ticket create` や `tm run start` がどの workspace/project/repo に作用するかが曖昧だと、cross-project safety が崩れます。

改善案:

- `ContextResolver` を ADR-00015/SP-016 に追加する。
- 優先順位は `explicit --workspace/--project/--repo`、env、cwd git remote mapping、profile default、interactive/error。
- mutating command は unresolved/ambiguous を fail-closed にする。
- `tm context show` で現在の解決結果、根拠、confidence、mutating allowed/blocked を表示する。

#### DI-20: CLI API capability token の binding が SecretBroker より弱い

API capability token は actor、principal、device、project、allowed_actions、audience、expires、jti、revoked などを持ちます。しかし SecretBroker が持つ operation fingerprint、one-time redeem、atomic claim と同等の binding が不足しています。

改善案:

- Tier 2/3 mutating command は `expected_request_fingerprint`、`payload_hash`、`approval_id`、`action_class`、`issued_run_id` を持つ。
- mutating call は atomic claim で one-time 化する。
- approval target、payload、resource scope がずれたら deny する。

#### DI-21: CLI ergonomics の受け入れ条件が薄い

`--json`、`--yaml`、`--quiet`、TTY 対応は出ていますが、実務 CLI に必要な paging、follow、filter、exit code、confirmation、dry-run、idempotency、machine-readable error schema が弱いです。

改善案:

- CLI UX contract を追加する。
- 最低限、`--project`、`--repo`、`--format`、`--no-pager`、`--follow`、`--since`、`--limit`、`--dry-run`、`--yes`、exit code taxonomy、machine-readable error schema を固定する。
- confirmation prompt は mutating/high-risk のみ。read/list/show は prompt しない。

#### DI-22: `tm` 名の衝突リスク

`tm` は短く便利ですが、alias/function/package との衝突余地があります。

改善案:

- canonical binary は `tmai` または `taskmanagedai` とし、`tm` は optional alias にする。
- SP-016 の smoke test は local PATH だけでなく shell alias/function も検出対象にする。

### 5. 複数プロジェクト、複数リポジトリ、worktree 操作性

#### DI-23: multi-repo 正常系がない

Project/Repository boundary と cross-project negative test は厚いです。一方、「frontend repo + backend repo」「docs repo + app repo」のような task をどう扱うかが未設計です。

改善案:

- P0/P0.1 で multi-repo を扱わないなら、`one ticket/run = one repository` と明記する。
- 扱うなら `ticket_repositories` または `run_repository_bindings` を追加する。
- orchestrator は parent ticket を child run per repo に分割し、PR も repo ごとに作る。
- CLI は `tm run create --repo A --repo B` のような明示指定を必須にし、推測で multi-repo mutating run を開始しない。

#### DI-24: worktree/bg job 運用が product flow に接続されていない

worktree setup、branch 命名、2-tier merge は運用文書として具体的です。ただし TaskManagedAI の UI/CLI から見ると、worktree や branch lane は外部手順のままです。

改善案:

- 最小でも `tm repo resolve`、`tm worktree plan`、`tm run attach-worktree`、`tm context show` を検討する。
- 実装しない場合も `docs/cli/README.md` に公式 SOP として残す。
- ContextSnapshot に `repo_state`、`worktree_path`、`branch`、`base_branch`、`dirty_summary` をどの条件で保存するか決める。

#### DI-25: artifact project boundary は cross-project CLI の前提として重要

Phase C では artifacts に project_id を materialize する議論があります。cross-project operation では、artifact がどの project/repo/run に属するかが UI/CLI の安全性に直結します。

改善案:

- P0.1 ではなく、cross-project CLI の設計前に artifact project boundary を確定する。
- 物理カラム追加を後回しにする場合でも、service guard と read model で `artifact_project_id` を明示する。

### 6. 設計 traceability、文書品質

#### DI-01: action_class drift

ADR-00009 と code は `task_write / repo_write / pr_open / secret_access / merge / deploy / provider_call` の 7 種を正本にしています。一方で DD-03/DD-04 などには `read/search` が action class として残っています。

改善案:

### BLOCK condition matrix

以下は、今回の deepening で明示的に追加した停止条件である。これらは「完全自律性を高める」方向の検討であっても緩めない。

| 条件 | 判定 | 必須 gate |
|---|---|---|
| 破壊的 shell、`git reset --hard`、`git clean`、広範囲 `chmod/chown` | BLOCK | 明示的なユーザー承認 + rollback plan |
| migration、DB schema、production data 操作 | BLOCK | ADR Gate + migration plan + rollback + tests |
| auth/authz、policy、approval、SecretBroker、provider、runner、MCP 書込境界の変更 | BLOCK | project-specific security gate + negative tests |
| provider call を policy decision / budget pre-spend / audited gateway なしに送信 | BLOCK | PolicyDecision + BudgetGuard + ComplianceGate enforce |
| AI/agent/service が approval decider になる経路 | BLOCK | human-only service/DB/test gate |
| Runner/MCP mutation gateway 完成前の command/file mutation 自律化 | BLOCK | Runner Boundary Gate + forbidden path/dangerous command tests |
| 外部 publish/deploy、public network exposure、Funnel/public IP CLI access | BLOCK | Network boundary ADR + user approval |
| `sprint_pack_ref` / ADR 参照が存在しないまま Sprint 着手 | BLOCK | canonical registry + planned/accepted status check |

### 自律性レベルの誤読防止

L3/L4 の `low-risk auto-allow` は、AI が approval decider になることを意味しない。approval が必要な action class では、decider は常に human-only であり、self-approval 禁止、stale invalidation、Approval 4 整合、policy decision、audit event は不変である。

P0/P0.1 の自律化で許せるのは、低リスクかつ allowlist 済みの artifact 生成、dry-run、read-only 調査、または policy profile が明示的に許す provider_call だけである。`repo_write`、`pr_open`、`secret_access`、`merge`、`deploy`、runner mutation、MCP mutation は、対応 gate が揃うまで自律化しない。

### P0 false-positive Exit を防ぐ追加 gate

P0 Exit は、次の条件を満たさない限り完了扱いにしない。

1. SP-012 の gold flow に ResearchTask / Claim / Evidence / generated Ticket が含まれる。
2. BL-0140 が Ticket 始点ではなく Deep Research-to-PR flow を代表ケースにする。
3. `generated ticket` 全件が Evidence/Claim/Decision lineage を持つ。
4. `citation_coverage`、`cost_per_completed_task`、review result が再計算可能な source を持つ。
5. P0 UI か `tm-lite` のどちらかで、承認待ち、失敗 run、blocked、next action を daily control plane として確認できる。
