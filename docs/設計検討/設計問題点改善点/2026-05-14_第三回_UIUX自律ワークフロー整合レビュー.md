# 2026-05-14 第三回 UI/UX・自律ワークフロー整合レビュー

目的: 前2回のレビュー成果物を全件読み直し、現行の設計文書・実装断面と照合して、特に UI/UX、初任者ワークフロー、自律承認、メインAIのタスク分配、計画/レビュー品質ループの不足を統合する。

今回の結論は明確である。TaskManagedAI の安全境界設計はかなり厚くなっているが、ユーザーが毎日使うタスク管理体験、初回利用者が安全に迷わず運用へ入る導線、メインAIがなぜそのタスクを選び、誰に渡し、どの品質ループで完了判定したかを説明する product surface はまだ弱い。完全自立性を高めるには、承認をなくすのではなく、どの低リスク操作が policy により自動 allow されたかを UI/CLI で説明可能にする必要がある。

## 3. UI/UX 重点レビュー

### DI-27: Dashboard が daily control plane ではなく health page になっている

前回の DI-08 は「P0 UI が daily control plane になっていない」という設計論点だった。今回の実装確認では、現行 Dashboard が実際に service health 表示に留まっていることを確認した。

改善案:

- Dashboard の主役を `System health` ではなく `Today / Inbox` にする。
- 最上段に `Needs decision`、`Running now`、`Blocked`、`Failed`、`Ready for review`、`Cost/risk alert` を置く。
- service health は右下または settings/ops view に退避する。
- 「朝一番に何をすべきか」が 5 秒で分かることを SP-009 の UI acceptance に入れる。

### DI-28: 初任者/初回利用者ワークフローがない

現在の設計は、開発者がすでに TaskManagedAI の安全モデルを理解している前提が強い。初任者、初回利用者、久しぶりに戻った利用者が、どの順番で project/repo/policy/autonomy profile を理解し、最初の安全なタスクを流すかが未設計である。

入れるべき場所:

1. First-run project setup
   - repo 接続、Tailscale/private boundary、GitHub App 状態、provider 設定、policy profile を確認する。
   - 初回は mutating action を開始せず、read-only diagnosis と sample plan だけを通す。
2. Task intake
   - 目的、対象 repo、期待成果物、許可する action_class、予算、期限、レビュー担当を guided form で入力する。
   - 初任者向けには `AIに調査だけさせる`、`計画だけ作らせる`、`Draft PRまで作るが承認必須` の3択を出す。
3. Plan review
   - action_class、risk、必要承認、費用上限、rollback、test plan を説明する。
   - `approve` だけではなく `request revision` と `ask why` を置く。
4. Run observation
   - 生ログではなく、現在状態、次に待っているもの、ユーザーがするべきことを表示する。
5. Post-run review
   - 何が変わったか、どの証拠に基づくか、どのテストで確認したか、次に残るリスクをまとめる。
6. CLI onboarding
   - `tmai init`、`tmai context show`、`tmai doctor`、`tmai run --dry-run` を最初の導線にする。

### DI-29: Approval Inbox は decision packet としてまだ薄い

Approval detail は証跡 hash と policy version を表示しており、前回より良い実装断面がある。ただし、ユーザーが「承認してよいか」を判断する材料としては、以下が不足している。

- diff summary と affected files
- expected command / tool calls
- required tests and current result
- cost estimate and budget remaining
- rollback plan
- stale reason and revalidation result
- reviewer artifact verdict
- request revision / ask why / split task / downgrade autonomy の選択肢

SP-009 は P0 must_ship を `approve/reject` の2ボタンに絞り、Edit を defer している。この判断は安全には妥当だが、UX と品質ループの観点では「Reject しか戻し方がない」状態になる。P0 で inline edit を入れないとしても、`request_revision` は別 action として定義すべきである。

### DI-30: Notifications が triage queue ではない

badge と list はあるが、タスク管理ツールとしては通知が行動可能単位になっていない。通知は `read/unread` ではなく、`required_action`、`severity`、`due_at`、`target_type`、`snoozed_until`、`dedupe_key`、`resolved_by` を持つべきである。

改善案:

- Notification を Inbox の一部として扱う。
- `approval_wait_ms` 悪化、run blocked、budget nearing limit、policy drift、review failed を triage lane に出す。
- mark read だけでなく `snooze`、`assign`、`open target`、`mark resolved` を設計する。

### DI-31: ナビとページ実体の差がある

Tickets、Agent Runs、Audit、Settings のリンクが存在するが、対応 page はまだない。これは現段階では実装途中として許容できるが、P0 UI の評価では「見えるが動かない」導線が UX の信頼を下げる。

改善案:

- P0 で未実装ページを出すなら `Coming soon` ではなく、対象データがまだ未実装であることと代替導線を明示する。
- P0 acceptance では nav link 全件の route 存在を Playwright で検証する。
- UI scope が未完成なら Dashboard から露出する項目を絞る。

### DI-39: Agent Runs workspace が未定義

今回のユーザー要求では、メインAIがタスクを分配した後に、各 run がどの状態で、誰が担当し、どの品質 gate を待っているかを確認できることが重要である。現行 UI は Agent Runs の nav label は持つが、run 一覧/詳細の page 実体はまだない。

Agent Runs workspace の必須表示:

- run state / phase / blocked reason
- owner actor / role / assignment reason
- linked ticket / parent run / child runs
- approval state / policy decision / autonomy profile
- last event / next action / retryability
- elapsed time / estimated and actual cost
- execution log summary lane / raw audit link
- reviewer artifact / conformance result

Verification:

- `failed`、`blocked`、`awaiting approval`、`completed` の fixture を用意する。
- Today/Inbox から該当 run detail へ到達できる。
- run detail から approval、ticket、artifact、raw audit へ戻れる。

### DI-40: Execution Log の summary lane 仕様が弱い

Execution Log は raw audit をそのまま見せるだけでは日次判断に使えない。一方で、summary だけでは監査性が落ちる。したがって、summary lane と raw audit lane の対応関係を持たせる必要がある。

summary lane 必須項目:

- `status`
- `why_stopped`
- `policy_or_validation_reason`
- `next_action`
- `affected_artifacts`
- `risk`
- `retryability`
- `owner`
- `raw_event_anchor`

raw event fixture から summary lane が生成でき、各 summary item が raw audit event に戻れることを acceptance に入れる。

### UI owner / route / state matrix

SP-009 へ反映する際は、少なくとも次の画面単位で owner、route/view、primary action、state model を固定する。

| View | Owner | Route / entry | Primary action | State model |
|---|---|---|---|---|
| Today / Inbox | Product UI | `/dashboard` | Decide next action | `empty / loading / ready / degraded / blocked` |
| Approval Inbox | Policy UI | `/approvals`, `/approvals/:id` | approve / reject / request_revision | `pending / approved / rejected / expired / invalidated / revision_requested` |
| Agent Runs | Runtime UI | `/dashboard/agent-runs`, `/dashboard/agent-runs/:id` | inspect / retry / cancel / resume-after-approval | AgentRun 16 states + blocked substate |
| Execution Log | Runtime UI | run detail timeline | explain stop and next action | `summary_lane / raw_audit_lane / redacted / unavailable` |
| Notification Queue | Ops UI | `/notifications`, Dashboard lane | open target / snooze / mark resolved | `created / surfaced / snoozed / actioned / escalated / resolved` |
| Newcomer Path | Onboarding UI | first-run checklist + guided task intake | create safe first dry-run | `not_started / setup_checked / guided_task_ready / dry_run_reviewed / first_decision_done` |

## 4. 旧オートモード的な「承認なし実行」は可能か

結論: 可能にしてよいのは、承認を「AIが代替する」形ではなく、Policy Engine が低リスク操作を事前に allow する形だけである。

### 許可できる形

`policy_profile=low_risk_auto_allow` のような profile が、以下をすべて満たす場合に限り、approval_requests を作らずに進める。

- 対象 action_class が profile に明示 allow されている。
- provider_call は ZDR / allowed data class / budget pre-spend / audit が揃っている。
- task_write は review_artifact または structured validation を通す。
- repo_write / pr_open / secret_access / merge / deploy は allow 対象にしない、または別 gate を必須にする。
- max_cost、max_duration、max_retries、max_children、max_depth、kill-switch がある。
- UI/CLI に `auto-allowed by policy profile` と理由が表示される。
- audit event と policy_decision が必ず残る。

### 禁止する形

以下は「旧オートモード」風に見えても採用してはいけない。

- AI agent が `approval_requests.decided_by_actor_id` になる。
- orchestrator が承認者になる。
- low-risk profile という名前で `repo_write` や `pr_open` を無制限に進める。
- provider call を budget / policy / compliance / audit なしで送る。
- human approval が必要な action を、reviewer agent の pass だけで実行する。
- stale diff / stale policy / stale artifact のまま承認を再利用する。

### UI での見せ方

承認を減らすほど、UI の説明責任は増える。ユーザーには次を表示する必要がある。

- 自動 allow された理由
- 適用された policy profile と version
- 実行された action_class と allowed_actions
- 最大予算と実消費
- どの gate を通過したか
- どの条件なら停止するか
- 次に人間が見るべき review artifact

## 5. メインAI / orchestrator の重み付けと分配

現行 ADR-00014 / SP-014 は、orchestrator の暴走を止める境界は強い。しかし「何を優先し、誰に渡し、どう品質を保証したか」をユーザーへ説明する設計がまだ不足している。

### DI-32: Task scoring / dispatch plan がない

追加すべき artifact:

- `task_score`
- `dispatch_plan`
- `assignment_reason`
- `review_plan`
- `quality_gate_plan`

推奨する重み付け軸:

| 軸 | 意味 | UI/CLI 表示 |
|---|---|---|
| impact | ユーザー価値、P0 Exit への寄与 | why now |
| urgency | due date、SLA、blocked chain | due / overdue |
| risk | action_class、secret/provider/repo mutation | required gate |
| complexity | estimated files, dependencies, unknowns | split needed |
| confidence | evidence coverage、test availability | confidence and gaps |
| readiness | ContextSnapshot、repo clean state、required inputs | ready/blocked |
| cost | token/currency/time budget | estimate / cap |
| skill fit | role、tool availability、provider suitability | assigned role |
| dependency | parent/child、blocking/blocked-by | critical path |

### DI-33: 分配結果の説明可能性がない

orchestrator は `dispatch hint` として role を使うが、role は権限ではない。この invariant は正しい。一方で、ユーザーから見ると「なぜ reviewer に渡したのか」「なぜ security_agent を追加したのか」「なぜ並列化したのか」が見えない。

改善案:

- child run 作成時に `assignment_reason` を必須にする。
- `risk >= high`、`action_class in repo_write/pr_open/provider_call/secret_access`、`budget_high`、`low confidence` なら reviewer/security/tester を追加する rule を設計する。
- UI の Agent Runs には role 名ではなく、担当目的、入力、期待成果物、停止条件を表示する。
- CLI は `tmai run plan --explain-dispatch` を持つ。

### DI-34: 品質ループが product artifact になっていない

今回ユーザーが求めている Quality Loop は、開発者の作業プロセスだけでなく TaskManagedAI の製品機能そのものにするべきである。つまり、タスクが渡されたときに計画をループし、レビューで指摘が出たら修正し、再レビューでゼロになるまで戻す、という状態遷移と証跡を UI/CLI/API に持つ必要がある。

必要な状態:

- `planning_draft`
- `planning_review`
- `planning_revision`
- `plan_fixed`
- `implementation`
- `implementation_review`
- `implementation_revision`
- `rereview`
- `conformance_check`
- `done`
- `blocked_needs_user`

必要な artifact:

- plan version
- review finding
- finding adoption decision
- resolution artifact
- rereview result
- conformance result
- harness incident
- stop/escalation reason

## 6. CLI / 複数プロジェクト運用

前回の A-CLI は妥当だが、今回の実装断面確認で、user CLI がまだないことを明確に書く必要がある。

### DI-35: CLI は設計はあるが実装入口がまだない

改善案:

- P0 で `tmai-lite` を入れるか、P0.1 まで UI only とするかを明示する。
- どちらの場合も `ContextResolver` の仕様は先に固定する。
- 複数 project/repo を扱う mutating command は、曖昧なら fail-closed。
- `tm` は alias 候補、canonical は `tmai` とする案を優先する。

### DI-36: terminal workflow に beginner mode が必要

CLI は熟練者向けに見えやすいが、実際には初任者にとって事故防止の入口にもなる。

推奨コマンド:

- `tmai init`: project/repo/provider/policy の状態確認
- `tmai context show`: cwd から解決された workspace/project/repo と根拠を表示
- `tmai doctor`: provider、GitHub App、Tailscale、policy、runner、secret readiness を診断
- `tmai ticket intake --guided`: guided intake
- `tmai run plan --dry-run`: 実行せず plan と risk を出す
- `tmai approvals list --explain`: 承認待ちと判断材料を表示

## 8. 推奨する次の修正順

1. SP-009 を `Today/Inbox + Approval decision packet + request_revision + Agent Run summary + beginner workflow` まで heavy 化する。
2. SP-012 に `ResearchTask -> Ticket -> Plan Loop -> Implementation -> Review Loop -> Draft PR -> Eval/Audit` を P0 gold flow として追加する。
3. ADR-00014/SP-014 に `TaskScoringService`、`DispatchPlan`、`assignment_reason`、`quality_gate_plan` を追加する。
4. ADR-00015/SP-016 に canonical CLI 名、ContextResolver、beginner commands、mode matrix を追加する。
5. DD-03/DD-04/DD-07 に `quality_loop_artifacts` と `harness_incidents` を追加する。

## 9. 完了判定

今回の第三回レビューは、前2回の結論を否定せず、現行実装断面で強弱を補正した。特に UI/UX については、設計上「作るべき」という段階から、現行 Dashboard/Approval/Notification/Nav の具体的な不足へ落とした。

次に進むときは、`改善アクションバックログ` の第三回追加 action を Sprint Pack/ADR 修正へ変換する。単に「承認をなくして速くする」方向へ進めるのではなく、低リスク auto-allow、品質ループ、説明可能な dispatch、初任者でも安全に使える UI/CLI をセットで扱うべきである。
