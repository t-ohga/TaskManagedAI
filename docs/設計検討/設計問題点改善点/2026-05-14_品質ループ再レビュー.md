## 再レビュー観点

### 1. 問題が P0/P0.1/P1 に切り分けられているか

確認結果: PASS

レポートでは、P0 で直すべき dogfooding slice、P0.1 の autonomy/UI-CLI parity、P1 の memory/multi-repo/local LLM を分けています。完全自律性についても、P0 で mutating autonomous work を許すべきではない点を明記し、段階的な autonomy level として整理しています。

### 2. 安全性を弱める改善案になっていないか

確認結果: PASS

改善案は、AI 出力の直接実行禁止、human-only approval、deny-by-default、policy decision 必須、runner sandbox 完成前の mutation block を維持しています。UI/CLI 操作性の改善も、ContextResolver の fail-closed、mutating command の明示 scope、API token の atomic claim など、安全境界を強める方向です。

### 3. UX 改善が抽象論に留まっていないか

確認結果: PASS

Today/Inbox control plane、Approval decision packet、2 層 Timeline、triage queue、KPI strip など、実装時に画面要件へ落とせる粒度で記載されています。SP-009 heavy 化に必要な route、primary action、state、owner、verification、stop gate も次アクションとして明示されています。

### 4. CLI/terminal 操作性が具体化されているか

確認結果: PASS

13 capability matrix、ContextResolver 優先順位、`tm-lite` の最小コマンド、CLI UX contract、multi-repo の採用/非採用判断、canonical binary 名の検討まで含めています。SP-006 が user CLI ではない点も明確に分離されています。

### 5. Cross-project / multi-repo の境界が曖昧なまま進まないか

確認結果: PASS

レポートは、cross-project negative test が厚い一方で multi-repo 正常系がない点を明示し、P0/P0.1 で `one ticket/run = one repository` と宣言する案、または `ticket_repositories` / `run_repository_bindings` を追加する案を提示しています。artifact project boundary も cross-project CLI の前提として扱っています。

### 6. 設計 traceability が改善対象として残っているか

確認結果: PASS

action class drift、missing Sprint Pack、Sprint Pack/status drift、canonical registry、design consistency ledger が明記されています。次の Sprint Pack/ADR 作成前に何を正本化するかが追える構成です。

## 残リスク

- 今回は設計文書レビューであり、コード修正やテスト実行は行っていません。
- 実装ファイルに対する安全性指摘は、静的確認と既存レビュー結果に基づくため、次の実装修正時には個別にテストで確認する必要があります。
- 外部プロダクトや provider 仕様の最新調査は今回の範囲外です。外部依存仕様を ADR/Sprint Pack に反映する場合は、その時点で再確認が必要です。

## 最終結論

このレポートは、次に作る Sprint Pack/ADR の入力として使用できる状態です。特に `action_class` 正本化、Deep Research E2E、欠落 Sprint Pack、SP-009 heavy 化、ADR-00015 の CLI matrix/ContextResolver、policy/runtime gate 未接続の Sprint Pack 化を、次の設計修正の上位タスクとして扱うべきです。
