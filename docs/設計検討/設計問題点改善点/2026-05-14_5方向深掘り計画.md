## 6. 第3回で補強した5方向計画

第3回では、同じ5方向を維持しつつ、UI/UX と自律ワークフローをより具体的な product requirement へ寄せる。詳細なファクトチェックは [第三回 UI/UX・自律ワークフロー整合レビュー](./2026-05-14_第三回_UIUX自律ワークフロー整合レビュー.md) を参照する。

| 方向 | 第3回で追加した問い | 必ず閉じる成果物 | 追加 stop gate |
|---|---|---|---|
| UX / Daily Operations | Dashboard が health page ではなく Today/Inbox になっているか。初任者が最初の安全な run まで迷わず進めるか | SP-009、UI route map、Playwright route test、guided intake spec | Dashboard が承認待ち/blocked/failed/next action を出せないなら P0 UI 完了不可 |
| Autonomy / Approval | no-approval が AI self-approval に誤読されないか。auto-allow の理由が UI/CLI に出るか | ADR-00014、SP-014、DD-04、policy_profile matrix | agent/orchestrator が decider になる設計は即 BLOCK |
| Orchestrator / Distribution | タスクがなぜ優先され、誰に分配され、どの reviewer を通るか説明できるか | ADR-00014、SP-014、TaskScoringService/DispatchPlan spec | assignment_reason なしの child run dispatch は BLOCK |
| Quality Loop | plan/review/revision/rereview/conformance が product artifact として残るか | DD-03、DD-07、review_artifacts、quality_loop_artifacts | finding 未解決や harness incident 未解決で done になるなら BLOCK |
| CLI / Cross-Project | `tmai context show` が project/repo/worktree を説明できるか。初任者向け dry-run があるか | ADR-00015、SP-016、docs/cli/README | ambiguous mutating command、canonical CLI 名未定義、route parity 未定義なら BLOCK |

### 第3回の acceptance 追加

- First-run / guided intake / plan review / run observation / post-run review / CLI onboarding のどこに初任者導線を入れるかが SP-009 または SP-016 に明記されている。
- `low_risk_auto_allow` は approval request を作らない policy effect であり、approval 決定ではないと明記されている。
- Orchestrator の dispatch には `task_score`、`dispatch_plan`、`assignment_reason`、`quality_gate_plan` がある。
- Review loop は「何回か回したから終了」ではなく、finding が resolved/deferred-with-owner/blocked のいずれかになるまで終了できない。
- Harness incident が出た場合は、未解決のまま clean evidence を出せない。
