## 2. Critical Gaps（P0 計画に組み込むべき抜け漏れ）

### G1. Sprint 0 相当の横断基盤が Sprint 1 に埋もれている

**問題**: 計画(仮).md の Sprint 1（Project Foundation）に「pnpm + uv + Docker Compose + Postgres + Redis + Next.js + FastAPI + worker の土台。CI、lint、typecheck、migration、seed、dev login」とあるが、以下の P0 横断基盤が陽に書かれていない。

- 観測（OpenTelemetry / Prometheus / Loki / Grafana）— 01 レポート §運用設計で必須化
- Sprint Pack テンプレート（`docs/sprints/_template.md`）の事前作成
- Backup / Restore drill（pg_dump + WAL archiving + restore test）— 01 レポート §構築手順 step 7
- E2E test 基盤（Playwright / httpx 等）の選定と起動
- Secrets 管理戦略（P0 は SOPS + age か Docker secrets か未確定）
- ネットワーク／公開方式の確定（後述 G2）
- Tailnet 構成（device approval / Tailnet Lock / tagged resources / grants）— 自作タスクツール無料運用構成.md §推奨構成
- CI から private staging への到達（Tailscale GitHub Action）— 01 レポート step 9

**推奨**: Sprint 1 を Sprint 0（プロジェクトハードニング）と Sprint 1（アプリ土台）に分離する、または Sprint 1 の acceptance criteria に上記を全て含めることを明示する。

### G2. ネットワーク／公開方式と CT log 対応が未記載

**問題**: 計画(仮).md は「Tailscale 閉域」とサマリで触れるだけで、以下が決まっていない。

- Tailscale Serve の HTTPS を有効化するか／するなら machine name を匿名化するか（CT log 漏洩対策）— 01 レポート §HTTPS と命名の注意点で強調
- Funnel は使わない（Serve のみ）の明文化
- Tailnet Lock vs device approval の選択（相互排他）
- Tailscale grants の最小化方針（自作タスクツール無料運用構成.md §強化施策）
- 商用化想定なら Tailscale Personal の "non-commercial" 制約に抵触する → 商用化フェーズで Tailscale Business または tsidp + 別 IdP に切り替える計画が必要

**推奨**: Sprint 0 または P0 設計文書に「ネットワーク境界設計」を独立節として書く。machine 命名規則（`taskhub-prod-NN` 等の中立名）を Sprint 0 で固定。

### G3. ContextSnapshot の必須項目が未仕様化

**問題**: 計画(仮).md は「ContextSnapshot」を AgentRuntime の構成要素として挙げるが、必須カラムが書かれていない。03 妥当性評価.md は以下を P0 必須としている。

- `prompt_pack_version`
- `policy_version`
- `repo_state`（SHA / branch / dirty フラグ）
- `tool_manifest`（その時点で使用許可された tools）
- `evidence_set_hash`
- `reasoning_ciphertext_ref`（OpenAI Responses `store:false` + encrypted reasoning carry-forward の参照）
- snapshot_kind（`input | pre_tool | post_tool | resume | final`）

**推奨**: Sprint 4（Agent Runtime）の Sprint Pack で ContextSnapshot のカラム定義を確定させる。再現性・監査性を担保する核なので妥協しない。

### G4. Tool Registry / Trust Tier が P0 から欠落

**問題**: 計画(仮).md は「ToolAdapter: read-only tools P0、MCP gateway P1 以降」と一行で済ませているが、03 妥当性評価.md は **P0 で `tool_registry`、`trust_tier=official|self_hosted|third_party|experimental`、read-only `search/fetch` と mutating tools の分離**を必須としている。

- MCP 認証仕様（OAuth 2.1 / audience validation / PKCE / token passthrough 禁止 / SSRF 対策）は P1 でも実装するため、P0 の tool registry がその受け皿になる必要がある
- AGENTS.md の「AI 出力を直接コマンド、SQL、ワークフロー、外部ツール操作へ接続しない」原則の実装基盤

**推奨**: Sprint 4 または 5 に「Tool Registry & Read-only/Mutating Gateway」を独立スプリントとして追加。

### G5. Policy / Approval engine の action class が粗い

**問題**: 計画(仮).md の Sprint 3（Policy And Approval）は「policy rules、approval requests、risk level、forbidden paths、write gate」とあるが、action class の分割が書かれていない。03 妥当性評価.md は以下を P0 必須としている。

- `read/search`
- `task_write`
- `repo_write`
- `pr_open`
- `merge`
- `deploy`
- `secret_access`

**推奨**: Sprint 3 の Sprint Pack で action class を確定し、各 class のデフォルト承認ポリシーを明文化する（read は自動 / write は差分確認 / merge/deploy/secret は必須承認）。

### G6. Eval の KPI（exit criteria）が未確定

**問題**: 計画(仮).md は「decomposition / coding / review / research / security / cost の薄い全領域 Eval」とあるが、**P0 完了の合否判定基準（exit criteria）が定義されていない**。03 妥当性評価.md は以下を提案。

- `acceptance_pass_rate`（受入条件達成率）
- `time_to_merge`（チケット→マージのリードタイム）
- `approval_wait_ms`（承認待ち時間）
- `citation_coverage`（claim に対する引用カバレッジ）
- `cost_per_completed_task`
- `approval_blocking`（ポリシーで止めた件数）

**推奨**: 計画(仮).md の §Assumptions の後に「P0 Exit Criteria」節を追加し、上記 KPI と最低達成水準を書く。Sprint 11（Eval Harness）で計測実装。

### G7. Deep Research 証拠の P0 仕様が薄い

**問題**: Sprint 10（Research/Evidence Foundation）は「ResearchTask、Claim、EvidenceSource、EvidenceItem の最小 DB/API」とあるが、03 妥当性評価.md が P0 で求める以下が未記載。

- `canonical_url`、`retrieved_at`、`published_at`
- `annotation_span`（出典の文字位置）
- `claim_hash`
- `relation = supports | contradicts | context`
- `provenance_json`（W3C PROV 準拠）
- 後続 P1 で `conflict_group_id`、`freshness_score`、`citation_render_mode`

**推奨**: Sprint 10 の Sprint Pack で上記カラムを確定。最低限 supports/contradicts/context の三値関係は P0 で入れる。

### G8. Cost Guard / Budget 強制停止の所属が曖昧

**問題**: 計画(仮).md は Agent Runtime に Budget を含めるが、以下が未仕様化。

- ticket 単位 budget cap、project 単位 cap、user 単位 cap の関係
- run timeout、再試行上限、並列数制限の policy 統合
- kill switch（緊急停止）の API / UI
- `unbounded_consumption`（OWASP LLM Top 10 2025）対策

**推奨**: Sprint 3（Policy）または Sprint 4（Agent Runtime）の Sprint Pack で budget hierarchy と kill switch を確定。

### G9. ハイブリッド前提（商用化準備）の schema 互換性が未定義

**問題**: ユーザー回答により「P0 は個人専用だが商用化準備」が確定。しかし計画(仮).md は「P0 では multi-tenant RLS は作らない」とのみ記載で、以下の schema 互換性確保が未定義。

- 全テーブルに `tenant_id` または `workspace_id` を最初から持たせるか（後付けで RLS 入れるなら必須）
- `users`, `workspaces`, `projects` の階層を最初から正しく組む（task機能検討.md §10 参照）
- `prompt_pack`, `policy_pack` を tenant ごとに version 管理できる構造
- secret reference model（個人運用は env 直、商用は Vault 参照に切替できる抽象化）

**推奨**: §Assumptions に「商用化準備のための schema 制約」節を追加し、`tenant_id` 必須カラムや `secret_ref` 抽象化を明文化する。

### G10. AI 出力サニタイズ／構造化検証が独立スプリントになっていない

**問題**: AGENTS.md は「AI 出力を直接コマンド、SQL、ワークフロー、外部ツール操作へ接続しない」を原則化している。計画(仮).md にこの原則の実装基盤（output validator、structured outputs / JSON schema 強制、redaction）が独立して位置付けられていない。

**推奨**: Sprint 5（Provider Adapter Foundation）の Sprint Pack で「全 provider 出力を JSON Schema で structured output 化、改行・引用符・マークダウンエスケープを redaction layer で通す」ことを必須にする。

## 3. Order / Sequencing Issues（順序・依存の懸念）

### O1. Sprint 3（Policy And Approval）の検証手段不在

Sprint 3 で policy/approval を作っても、Sprint 4（Agent Runtime）と Sprint 7（Docker Isolated Runner）と Sprint 8（GitHub Draft PR Flow）が揃わないと **承認対象の write 操作が存在しない**。

**推奨**: Sprint 3 では「Mock Run（dry run / 仮想 write）に対する承認 flow」までを acceptance criteria に入れ、実際の write は Sprint 7-8 で再検証する設計にする。

### O2. Sprint 9（P0 UI）が後ろすぎる

UI が Sprint 9 まで存在しないと、Sprint 3-8 の検証が API レベルでしか出来ない。Approval Inbox は Sprint 3 の動作確認に必要。

**推奨**: Sprint 1 で "minimal admin UI shell"（auth + nav + ticket list の骨組み）を作り、Sprint 3 で Approval Inbox、Sprint 4 で Agent Runs を順次足す**段階的 UI 構築**にする。

### O3. End-to-End Review（Sprint 12）が "確認" で終わっている

「Ticket 作成から計画、承認、runner、Draft PR mock、本格 Eval、監査確認まで通す」とあるが、**P0 exit criteria で何を満たせば P0 完了か**が不明。

**推奨**: Sprint 12 を「P0 受入試験」と名付け、G6 の KPI と Sprint 1-11 の acceptance criteria を全て通すゴールドフロー（30-50 件の private gold tasks）を定義する。

## 4. Ambiguity（曖昧で確認すべき点）

### A1. Worker / Queue のスタック未確定

01 レポートでは BullMQ + Redis または軽量 Python worker と提案。計画(仮).md は「pnpm + uv + ... + worker」と書くが Node 系か Python 系か未指定。

**推奨**: Sprint 1 の Sprint Pack で確定。FastAPI 中心なら arq / RQ / Celery、TypeScript 中心なら BullMQ。混在は避ける。

### A2. CLI Artifact Orchestration（Sprint 6）の実装スタイル

「`codex exec` / `claude -p` 等を直接密結合せず、Markdown/JSON artifact 経由で制御」とあるが、CLI の起動主体（worker process / spawned subprocess / docker run）が未定。

**推奨**: Runner Adapter（Sprint 7）と一体で設計する。CLI は subprocess として spawn、stdout/stderr → artifact 化、exit code → run status とする方針を Sprint 6 Pack で明文化。

### A3. GitHub App の権限スコープ

Sprint 8 は「GitHub App、branch 作成、push、Draft PR 作成、CI status 取得」だが、Webhook HMAC 検証、installation token のローテーション、複数 repo 管理が未記載。

**推奨**: Sprint 8 Pack で App permissions を `Contents: read/write`, `Pull requests: read/write`, `Metadata: read`, `Actions: read` に限定する明記。

### A4. Notification Adapter の P0 扱い

「P0 は未実装、Slack/Email/Discord 等を後続追加」とあるが、Approval 待ち通知が無いと P0 で運用ループが回らない（個人運用でも自分への通知は必要）。

**推奨**: P0 は最低でも「メール 1 通 / 既存 Discord MCP / VS Code 通知 のいずれか」を Sprint 3 か Sprint 4 で入れる。

### A5. ADR の運用基準

「high-risk 変更は ADR も作る」とあるが、何を high-risk と判断するかが未定義。

**推奨**: G5 の action class と連動させ、`merge / deploy / secret_access / DB schema 変更 / auth 変更` を ADR 必須に固定する。

## 5. Risk（リスク提示）

### R1. P0 機能数が多すぎる懸念

12 スプリント × 各 Sprint Pack 必須ゲート = 12 個の事前ゲート + 12 個の Sprint Review = 24 ドキュメント。個人運用前提なら過剰になりやすい。

**緩和**: Sprint Pack を「軽量 Pack（テンプレ埋め）」と「重量 Pack（ADR 込み）」の 2 段にする。Sprint 1, 6, 9 など UI/土台系は軽量、Sprint 3, 4, 7, 8 など権限/実行/外部連携系は重量。

### R2. ハイブリッド前提のオーバーエンジニアリング

「商用化準備」と称して全テーブルに tenant_id を入れると、個人運用フェーズで認証コストが先に立つ。

**緩和**: P0 の `tenant_id` は **default 値 1 で固定**し、RLS は P1 で有効化、API も P1 で multi-tenant 化、と段階を分けて書く。

### R3. Eval の合成 fixture と実 repo 抽出のバランス

「合成 fixture と実 repo 抽出ケースを併用」だが、合成 fixture の作成コストが見積もられていない。

**緩和**: P0 では 30-50 件の private gold task（自社 repo 由来）を最優先で集め、合成 fixture は P1 以降。

### R4. Tailscale Personal の商用化制約

商用化時に Personal プランの "non-commercial" 制約に抵触。tsidp は experimental。

**緩和**: 商用化の予約として「Personal → Business + tsidp/IdP 切替」を移行計画に明記。
