# 02 領域別改善案

作成日: 2026-05-14

## 1. Product / Gold Flow

### 問題

現在の P0 完了判定は、Deep Research を通らずに Ticket-to-PR flow だけで完了できる余地があります。これは TaskManagedAI の中核価値である「証拠、判断、承認、実行ログ、コスト、レビュー結果付きの AI-native 開発管理」を検収しない false-positive です。

### 修正案

- SP-012 の代表 gold flow を `ResearchTask -> Claim/Evidence -> generated Ticket -> Plan Loop -> Draft PR -> Eval/Audit` に変更する。
- Research なしの Ticket-to-PR は smoke として残すが、P0 Exit の代表ケースにはしない。
- `generated ticket` 全件が Evidence / Claim / Decision lineage を持つことを Hard Gate 化する。
- `citation_coverage`、`grounded_answer_rate`、`cost_per_completed_task`、review verdict を再計算可能にする。

### 反映元

- 設計問題点: DI-02、DI-07、DI-10、DI-11、DI-38、A-PROD、A-QL
- Gemini: GMG-F-007、F-008、F-009、F-010、F-012、F-015、F-026、F-027
- Realtime: transcript/event log UI、eval fixture plan

## 2. UI / Daily Operations

### 問題

Dashboard が service health 中心で、Today/Inbox control plane になっていません。Approval は `approve/reject` に寄り、Notification は triage queue ではなく、Agent Runs workspace は未定義です。

### 修正案

| UI | 修正 |
|---|---|
| Dashboard | `Needs decision`、`Running now`、`Blocked`、`Failed`、`Ready review`、`Cost/risk alert` を最上段に置く |
| Approval | decision packet 化し、`request_revision` を approve/reject と別 action にする |
| Agent Runs | run state、assignment reason、approval state、blocked reason、cost、retryability、child runs を一覧/詳細で出す |
| Execution Log | summary lane と raw audit lane を分離し、summary item から raw event へ戻れるようにする |
| Notifications | `required_action`、`severity`、`due_at`、`snooze`、`dedupe`、`resolved_by` を持つ queue にする |
| Newcomer Path | first-run setup、guided intake、plan review、dry-run、post-run review、CLI onboarding を用意する |

### 検証

- Playwright route existence test。
- empty / loading / ready / degraded / blocked state。
- redaction snapshot。
- request_revision -> old approval invalidation -> revalidation flow。

## 3. Quality Loop / Review Artifact

### 問題

Quality Loop が開発手順としては存在していても、TaskManagedAI の product artifact としてまだ十分に一級化されていません。review report が自然文に寄ると、採否、解決、再レビュー、conformance が後から機械的に追えません。

### 修正案

- `quality_loop_artifacts` か既存 artifact schema extension を作る。
- `review_results` は `verdict`、`findings_count`、`severity`、`target_artifact_id`、`target_hash`、`required_revision`、`resolved_by_artifact_id` を持つ。
- finding は `open / adopted / resolved / rejected / deferred` を持つ。
- non-INFO defer は owner、impact、resume condition がないと fixed にできない。
- harness incident は Quality Loop の内部問題として分離し、未解決なら clean evidence を禁止する。

### UI 表示

- Approval / Agent Runs / Execution Log で review finding と resolution を見せる。
- 「何回回したか」ではなく「未解決 finding が 0 か」で完了を表示する。

## 4. Autonomy / Orchestrator

### 問題

完全自律性を高めたい一方で、承認なし実行が AI self-approval と誤読される危険があります。また、orchestrator が何を優先し、なぜ child run に分配し、どの reviewer を通したかの product surface が弱いです。

### 修正案

#### Auto-allow semantics

- `low_risk_auto_allow` は approval ではなく Policy Engine effect。
- approval request は作らず、decider も存在しない。
- UI/CLI に policy profile、policy version、allowed action、cost cap、stop condition、auto-allowed reason を表示する。

#### Dispatch explainability

追加 artifact:

- `task_score`
- `dispatch_plan`
- `assignment_reason`
- `review_plan`
- `quality_gate_plan`

推奨 scoring axis:

| 軸 | 意味 |
|---|---|
| impact | P0 Exit やユーザー価値への寄与 |
| urgency | due / SLA / blocked chain |
| risk | action_class、secret、provider、repo mutation |
| complexity | files、dependencies、unknowns |
| confidence | evidence coverage、test availability |
| readiness | ContextSnapshot、repo clean state、required inputs |
| cost | token/currency/time budget |
| skill fit | role/tool/provider suitability |
| dependency | parent/child、blocking relation |

Stop gate:

- assignment_reason なしの child run dispatch は BLOCK。
- high risk / low confidence / provider_call / repo_write を単独 implementer に渡さない。

## 5. CLI / Cross-Project

### 問題

User CLI は設計上は重要ですが、まだ実装入口がありません。`tm`、`taskhub`、`tmai` の命名も揺れ、ContextResolver が未固定です。

### 修正案

- canonical binary は `tmai`。
- `tm` は optional alias。
- `taskhub` は host/admin CLI と user CLI を混同しない。
- ADR-00015 に13 capability matrix を追加する。
- SP-016 は ContextResolver、mode matrix、UX contract、API token atomic binding を acceptance にする。
- project auto-discovery は CLI ContextResolver と隣接するが、memory は ContextSnapshot を置き換えない supplement として扱う。
- project/repo boundary をまたぐ memory retrieval は deny-by-default にし、source project、freshness、confidence、redaction state を audit へ残す。

ContextResolver:

| 入力 | 扱い |
|---|---|
| explicit flag | 最優先 |
| env | second |
| cwd git remote | confidence と根拠を表示 |
| profile default | fallback |
| conflict | read-only は説明、mutating は fail-closed |

Project auto-discovery / memory boundary:

| 論点 | 扱い |
|---|---|
| project registry | repo / worktree / remote mapping の正本候補 |
| local memory | ContextSnapshot の補助情報。正本状態は上書きしない |
| cross-project retrieval | explicit allowlist と audit がない限り deny-by-default |
| memory-derived suggestion | source project、freshness、confidence、redaction state を表示 |

## 6. Provider / Tool / Runner Boundary

### 問題

Gemini / OpenAI sample は tool calling、MCP、managed execution、Realtime tools などを示しますが、これらを TaskManagedAI に直接つなぐと AI output boundary を壊します。

### 修正案

#### ProviderAdapter

追加/明確化する field:

- `api_or_feature`
- `model_requested` / `model_resolved`
- `schema_ref` / `schema_fingerprint`
- `payload_data_class`
- `provider_request_fingerprint`
- `tool_manifest_ref`
- `retry_policy_ref`
- `idempotency_key`
- `safety_metadata_ref`
- `retry_observation`

`allowed_data_class` は caller input にしない。server が artifact/context と Provider Matrix から算出する。

#### Tool Registry

- MCP server allowlist。
- tool schema hash。
- risk level / action class。
- approval requirement。
- sandbox profile。
- tool output trust level = `untrusted_content`。

#### Runner / managed execution

Gemini Code Execution / Computer Use や Realtime MCP direct execution は、Runner/Tool Gateway 外では reject。将来採用する場合は、sandbox、artifact capture、approval、safety acknowledgement、audit が必須です。

## 7. Evidence / RAG / Eval

### 問題

Research/Evidence/Eval は TaskManagedAI の差別化中核ですが、現状では generated ticket lineage、citation coverage、retrieval ranking eval、review result artifact が十分に一体化していません。

### 修正案

- local-first retrieval を採用し、Vertex / managed search は adapter candidate に留める。
- `SearchRun`、`EvidenceSearchHit`、`GroundingSupport`、`RankingPolicyVersion`、`RetrievalEvalRun`、`EmbeddingSourceVersion`、`UrlSourceSnapshot` の landing を既存 schema / artifact extension / defer に分類する。
- metrics は `recall@k`、`precision@k`、`ndcg@k`、`citation_coverage`、`grounded_answer_rate`、`tool_trajectory_match`、`approval_gate_compliance`、`secret_canary_no_leak`。
- synthetic Q&A / benchmark item は human validation 前に acceptance fixture にしない。
- URL Context は snapshot/hash/trust policy/citation verifier が揃うまで evidence 正本にしない。

## 8. OpenAI Realtime / InteractionGateway

### 問題

Realtime は自然な低遅延体験を作れますが、browser event、audio retention、sideband、MCP、BudgetGuard、Provider Matrix を同時に拡張する高リスク surface です。

### 修正案

- Realtime は `ProviderAdapter` ではなく `InteractionGateway` で扱う。
- OpenAI Realtime adapter は `RealtimeInteractionAdapter`。
- transport は `RealtimeSessionBridge`。
- P0 runtime には入れない。
- 先に text-only low-latency intake と STT -> ProviderAdapter -> TTS を同一 fixture で比較する。
- Realtime は unified `/v1/realtime/calls` + backend sideband を第一候補にし、client secret flow は代替候補にする。

必須 gate:

- Provider Matrix rows。
- SecretBroker-mediated session creation。
- canonical session config。
- browser event allowlist。
- sideband server control。
- Output Validator fail-closed。
- data retention / consent。
- BudgetGuard / modality cap。
- kill switch。

## 9. Gemini / Google Cloud Managed Feature

### 問題

Gemini sample は多くの high-value pattern を持ちますが、Google managed services を TaskManagedAI core runtime にすると、source of truth、provider compliance、retention、cost、region、audit が外へ逃げます。

### 修正案

| Surface | 判断 |
|---|---|
| structured output / function schema | narrow ProviderAdapter mapping として adopt |
| retry / timeout | provider-neutral reliability policy として adopt |
| safety ratings / attack examples | SafetyMetadata + negative fixture として adopt |
| hybrid search / custom ranking | local-first concept として adopt |
| Vertex Search / Agent Platform / Memory Bank | P0 core にはしない |
| Code Execution / Computer Use | direct use reject |
| URL Context | source snapshot まで defer |
| Batch Prediction | async job ledger まで defer |
| Multimodal Live API | P1+ defer high-risk |
| Model Optimizer | resolved model trace まで defer |
