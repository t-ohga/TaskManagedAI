# 07 矛盾と正本優先順位

作成日: 2026-05-14

## 1. 目的

3系統の文書を統合すると、同じ概念が別名・別粒度で出る箇所があります。次の上流修正で迷わないよう、現時点の優先順位を固定します。

## 2. 正本優先順位

| 論点 | 正本 | 補足 |
|---|---|---|
| action_class | ADR-00009 + backend code/tests の7種 | `read/search` は action_class ではなく Tool Registry allowed action |
| P0 gold flow | SP-012 + P0 backlog BL-0140 を修正して正本化 | このまとめは proposal。SP-012 未修正なら P0 Exit はまだ未解決 |
| Sprint id / backlog id | `docs/sprints/README.md` に作る canonical registry | global P0 `BL-*` と Sprint-local `BL-*` は namespace 分離 |
| no-approval | Policy Engine auto-allow | AI approval / orchestrator approval ではない |
| CLI name | `tmai` を canonical 候補、`tm` は optional alias | `taskhub` は admin/host CLI として混同しない |
| Realtime boundary | `InteractionGateway` | `ProviderAdapter` 置換ではない |
| Gemini managed feature | feature 単位 Provider Matrix | `gemini/generate_content` public-only row を拡張解釈しない |
| Evidence truth | TaskManagedAI snapshot/hash/citation verifier | provider grounding / URL metadata は seed |
| Context truth | ContextSnapshot | provider cache、Realtime session、Memory Bank は metadata/ref |
| Review completion | findings zero + resolved/deferred-with-owner + conformance | 回数だけでは完了にしない |

## 3. 衝突しやすい事項

### 3.1 `SP-010` / `SP-011` が未作成または揺れている

扱い:

- Gemini / Realtime / 設計問題点の各文書は `SP-010_research_evidence`、`SP-011_eval_harness` を受け皿として参照する。
- ただし、現時点の参照だけでは実装準備完了ではない。
- 次の修正では、作成するか、既存 Sprint に統合するか、P0 から明示除外する。

### 3.2 `BL-*` の衝突

扱い:

- `BL-0113` などは global P0 backlog を指す文脈がある。
- Sprint Pack-local backlog と衝突し得る。
- registry では `P0-BL-*` と `SPxxx-BL-*` のように namespace を分ける。

### 3.3 `read/search` action class drift

扱い:

- policy action_class は7種。
- read/search は Tool Registry allowed action、capability、risk level、network policy として扱う。
- docs に `read/search` が action_class として残るなら drift。

### 3.4 P0 UI の scope

扱い:

- system health dashboard だけでは P0 UI 完了にしない。
- P0 の minimum は Today/Inbox、Approval decision packet、Agent Runs、Execution Log summary、Notification queue のうち、少なくとも daily control plane を成立させること。
- 未実装 nav は route existence / disabled-deferred state を明示する。

### 3.5 Approval と request revision

扱い:

- P0 inline edit defer は維持してよい。
- ただし `reject` しか戻し方がない状態は Quality Loop として弱い。
- `request_revision` は approve/reject と別 action として設計する。

### 3.6 Realtime の扱い

扱い:

- P0 runtime は BLOCK。
- UI pattern は PASS。
- P0.1 prototype は WARN with gates。
- first topology は unified `/v1/realtime/calls` + backend sideband。
- client secret flow は secondary。
- browser-supplied model/tools/instructions/session config は reject。
- voice "OK" は approval ではない。

### 3.7 Gemini の扱い

扱い:

- structured output / tool schema / retry / safety metadata / local-first retrieval pattern は採用可能。
- Vertex Search、Agent Platform、Memory Bank、Code Execution、Computer Use、URL Context、Batch、Live API、Model Optimizer は別 feature gate。
- `gemini/generate_content` public-only entry は broad permission ではない。

### 3.8 Quality Loop の扱い

扱い:

- 開発手順だけではなく product artifact にする。
- review finding、adoption decision、resolution、rereview、conformance、harness incident を保存する。
- open finding や unresolved harness incident のまま clean evidence を出さない。

## 4. 次の設計修正で使う判断

| 判断が必要な場面 | 推奨 |
|---|---|
| いきなり Realtime 実装を入れるか | 入れない。InteractionGateway ADR と fixture が先 |
| Gemini Code Execution を便利だから使うか | 使わない。Runner/Tool Gateway gate が先 |
| P0 Exit を Ticket-to-PR で通すか | 通さない。Research-to-PR を代表 flow にする |
| Approval UX は approve/reject だけでよいか | request_revision を別 action として設計する |
| no-approval を自動承認として見せるか | 見せない。policy auto-allow として理由を表示する |
| CLI は `tm` でよいか | canonical は `tmai`、`tm` は alias 候補 |
| Multi-repo を推測で扱うか | 扱わない。明示指定か parent/child split |
