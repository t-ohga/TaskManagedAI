## 5. 守るべき不変条件

- このまとめは doc-only synthesis。次の run で正本候補を修正するまで、P0 完了条件が直ったとは扱わない。
- AI output を shell、SQL、workflow、MCP、provider tool call、repo mutation へ直接つながない。
- Approval decider は human-only。agent / orchestrator / service / provider は decider にならない。
- `low_risk_auto_allow` は approval ではなく Policy Engine effect。
- SecretBroker は raw secret 非露出、one-time claim、operation fingerprint、approval binding を守る。
- Gemini Code Execution / Computer Use / Realtime MCP direct execution は Runner/Tool Gateway 外では採用しない。
- Realtime は `ProviderAdapter` 置換ではなく `InteractionGateway` の intake boundary として扱う。
- Memory は ContextSnapshot を置き換えない。project/repo boundary をまたぐ retrieval は deny-by-default。
- `read/search` は policy action_class ではなく、allowed action / Tool Registry 側へ寄せる。
- この run 自体は `mode=no_hooks` / `hooks_enabled=false` で閉じている。次の実装 run ではこの active run の hook enforcement に依存せず、対象 run の write scope、rollback、verification、approval 条件を明示してから進める。

## 6. 各 run の完了条件

各 Quality Loop run は、少なくとも次を満たすまで完了にしない。

- 対象正本と write scope が明示されている。
- 変更理由、採用しなかった案、defer 条件、rollback が記録されている。
- `04_出典トレースマトリクス.md` の該当 DI / action ID へ逆参照できる。
- acceptance、verification、stop gate が自然文だけでなく、表または fixture / test 名で追える。
- high-risk scope に入る場合は、security / provider / runner / approval の gate が別項目でレビューされている。
- `/Users/tohga/.codex/skills/quality-loop-orchestrator/scripts/validate_quality_loop.py` が `ok: true` / `issues: []`。
- unresolved `BLOCK` / `WARN` / `ERROR` finding、harness incident、subagent job が残っていない。

## 7. 触ってはいけない範囲

この引き継ぎから直接触ってはいけないもの:

- production DB、migration、seed、実データ。
- auth / authorization / approval service のコード。
- provider credentials、`.env`、SOPS secret、SecretBroker secret payload。
- Runner / MCP mutation gateway の実装コード。
- Realtime / Gemini / provider SDK の runtime wiring。
- external publish、deploy、public exposure。

これらが必要になった場合は、別 Quality Loop で write scope、rollback、negative test、user approval 条件を固定してから進める。
