## 2. Phase 0: 実装前の正本化

### 2.1 Sprint Pack registry

直すべき問題:

- `SP-008`、`SP-010`、`SP-011`、`SP-011.5`、`SP-0045_tool_registry` などが参照されるが、存在・名称・P0範囲が揺れている。
- Gemini intake でも `SP-010_research_evidence` / `SP-011_eval_harness` が重要な受け皿として出ているが、取り込み文書だけでは Sprint Pack 正本にならない。

修正案:

- `docs/sprints/README.md` に canonical registry を作る。
- 各 Sprint Pack について `id`、`file`、`status`、`p0_scope`、`depends_on`、`backlog_refs`、`missing/deferred reason` を記録する。
- `P0-BL-*` と Sprint-local `BL-*` の namespace を分ける。

Stop gate:

- `sprint_pack_ref` が存在しないまま当該 Sprint を着手しない。

### 2.2 action_class drift

正本:

```text
task_write / repo_write / pr_open / secret_access / merge / deploy / provider_call
```

修正案:

- `read/search` を policy action_class から外す。
- `read/search` は Tool Registry の allowed action、risk level、capability として表現する。
- docs/code/tests の enum drift check を追加する。

Stop gate:

- `read/search` が policy action_class として残る設計文書があるなら、高リスク Sprint 着手を止める。

## 3. Phase 1: P0 gold flow

P0 Exit は、Ticket-to-PR の smoke だけでは通さない。TaskManagedAI の代表ケースは Deep Research-to-PR です。

必須 flow:

```text
ResearchTask
-> Claim / Evidence / citation coverage
-> research decision
-> generated Ticket
-> plan loop
-> approval / request revision
-> implementation / Draft PR
-> review loop
-> Eval / Audit / cost
```

必要な schema / artifact:

| 項目 | 最小 landing |
|---|---|
| `research_decisions` | decision、accepted/rejected/deferred、decider、evidence_set_hash |
| `research_generated_tickets` | ResearchTask / Decision / Ticket lineage |
| `evidence_to_acceptance_criteria` | AC と Evidence / Claim の依存 |
| `GroundingSupport` | claim span、source snapshot、citation verifier status |
| review artifact | verdict、findings、target hash、resolution、rereview |
| cost ledger | pricing_version、billable_usd、cost_source、ticket_id、provider_request_fingerprint |

Stop gate:

- Research を通らない Ticket-to-PR だけで P0 Exit を完了扱いにしない。
- generated Ticket が Evidence / Claim / Decision lineage を持たない場合は P0 gold flow fail。

## 4. Phase 2: SP-009 UI heavy 化

現状確認では Dashboard は health card 中心で、日次運用の入口ではありません。SP-009 は次の must_ship に変えます。

| View | Must ship |
|---|---|
| Dashboard / Today Inbox | Needs decision、Running、Blocked、Failed、Ready review、Cost/risk alert、next action |
| Approval Inbox | action_class、risk、diff summary、policy reason、evidence、cost、rollback、stale state、request_revision |
| Agent Runs | state、phase、blocked reason、ticket、approval、assignment reason、cost、next action、retryability、child runs |
| Execution Log | summary lane と raw audit lane、`raw_event_anchor` |
| Notification Queue | required_action、severity、due、snooze、dedupe、target route、resolved state |
| Newcomer Path | first-run setup、guided intake、plan review、run observation、post-run review、CLI onboarding |

Acceptance:

- Route existence / disabled-deferred state を Playwright で確認する。
- raw provider payload、client secret、secret-like value、tool args は redacted-by-default。
- Dashboard が system health だけなら P0 UI 完了不可。

## 5. Phase 3: Quality Loop artifact 化

ユーザーが求める「何度もループして完璧にする」は、開発手順だけでなく TaskManagedAI の product concept として扱います。

必要な状態:

```text
planning_draft
planning_review
planning_revision
plan_fixed
implementation
implementation_review
implementation_revision
rereview
conformance_check
done
blocked_needs_user
```

必要な artifact:

- plan version
- review finding
- adoption / rejection / defer decision
- resolution artifact
- rereview result
- conformance result
- harness incident
- stop / escalation reason

Stop gate:

- finding open のまま done にできない。
- non-INFO defer に owner / resume condition がない場合は fixed にできない。
- unresolved harness incident がある場合は clean evidence を出せない。

## 6. Phase 4: Autonomy / safety runtime gate

自律性は「承認をなくす」方向ではなく、「低リスクで policy が許可した操作だけを説明可能に自動化する」方向で上げます。

必須修正:

| 項目 | 必須条件 |
|---|---|
| PolicyDecision | provider/secret/repo/runner/tool の外部送信・mutation の前に必須 |
| BudgetGuard | max token/cost/wall-clock/retry/repair/parallel/session を pre-spend で評価 |
| Provider gateway | `evaluate` と `enforce` を分離せず、audited gateway に一本化 |
| SecretBroker | operation fingerprint、approval_id、payload_hash、one-time claim |
| Runner/MCP gateway | forbidden path、dangerous command、network policy、artifact redaction |
| Auto-allow | approval request を作らない policy effect。AI approval ではない |

Stop gate:

- agent / orchestrator が approval decider になる経路は即 BLOCK。
- low-risk の名前だけで `repo_write`、`pr_open`、`secret_access`、`merge`、`deploy` を許さない。

## 7. Phase 5: CLI / ContextResolver

User CLI はまだ実装入口がないため、先に仕様を固定します。

推奨:

- canonical binary は `tmai`。
- `tm` は optional alias。
- `taskhub` は admin/host CLI と user CLI を混同しない。

最小 P0 / P0.1 command:

- `tmai init`
- `tmai context show`
- `tmai doctor`
- `tmai ticket intake --guided`
- `tmai run plan --dry-run`
- `tmai approvals list --explain`
- `tmai run show --follow`

ContextResolver 優先順位:

1. explicit `--workspace` / `--project` / `--repo`
2. env
3. cwd git remote mapping
4. profile default
5. interactive select or fail

Project auto-discovery / memory boundary:

- project registry / repo mapping / worktree identity を P0.1 で最小仕様化する。
- memory は ContextSnapshot を置き換えず、source project、freshness、confidence を持つ supplement として扱う。
- project/repo boundary をまたぐ memory retrieval は deny-by-default にする。

Stop gate:

- mutating command で unresolved / ambiguous context は fail-closed。
- multi-repo run は推測しない。明示指定または parent/child split が必要。
- memory 由来の提案だけで project/repo/worktree を確定しない。

## 8. Phase 6: Provider / Realtime / Gemini feature intake

P0 の本線を閉じるまで、外部 feature は設計 input として扱います。

先にやること:

- Provider Matrix を feature 単位にする。
- Gemini `generate_content` public-only entry を managed search / code execution / URL context / live / batch に拡張解釈しない。
- OpenAI Realtime は `InteractionGateway` で扱い、`ProviderAdapter` 置換にしない。
- text-only baseline と STT chained voice pipeline を Realtime より先に評価する。

Prototype admission:

- task draft acceptance が baseline より改善。
- p95 latency または correction count が STT chained より改善。
- browser forbidden event count = 0。
- approval misunderstanding count = 0。
- cost cap 超過なし。
- audit replayability が baseline 同等以上。
