# プロダクト要求定義

## 1. Vision

### PRD-00-VISION-1: 中核コンセプト

TaskManagedAI は、Deep Research から実装 PR までを、証拠、判断、承認、実行ログ、コスト、レビュー結果とともに管理する AI-native な開発タスク管理ツールである。

単なるタスク管理ツールでも、単なる AI coding runner でもなく、調査、意思決定、実装、レビュー、運用をつなぐ開発 OS として扱う。調査テーマを登録すると、AI が関連情報を調査し、証拠付きで結論をまとめ、実装可能なチケットへ分解し、リポジトリ文脈を読んで実装計画を作り、人間承認後に runner と GitHub Draft PR 作成まで進める。

すべての判断、引用、AI 入出力、tool 呼び出し、承認、コスト、CI 結果、レビュー結果は監査可能な artifact と event として残す。これにより、AI の作業を「チャットで消える作業」ではなく、再現可能で評価可能な開発プロセスとして管理する。

### PRD-00-VISION-2: P0 の到達点

P0 は個人専用、Tailscale 閉域、単一 VPS を前提に、AI タスク実行の核を安全に成立させる段階である。P0 で実現する価値は、個人開発者が TaskManagedAI 自体を dogfooding しながら、チケット作成、計画、承認、AI run、runner 実行、Draft PR mock、Eval、監査確認までを一気通貫で検証できることである。

P0 では多人数 SaaS としての完成度を狙わない。ただし、schema、API 境界、権限境界、secret 抽象化、adapter、event、artifact、tenant 境界 invariant は、将来のチーム運用と商用化へ低コストで移行できるよう最初から準備する。

### PRD-00-VISION-3: 安全な AI 実行の原則

AI 出力は、直接コマンド、SQL、workflow、外部ツール操作へ接続しない。AI は計画、patch、レビュー、Evidence、Policy 判定などの artifact を生成し、アプリ側が構造化検証、policy 判定、人間承認、runner sandbox を通して採否を決める。

このプロダクトの競争力は、AI に広い権限を渡すことではなく、AI の提案を安全境界、証拠、承認、評価、監査の上に載せて実行可能にすることである。

## 2. 対象ユーザー / ユースケース

### PRD-00-USER-1: P0 対象ユーザー

P0 の対象ユーザーは、TaskManagedAI を自分専用に運用する個人開発者である。

このユーザーは、Tailscale に参加した自分の端末から VPS 上の TaskManagedAI にアクセスし、個人の開発タスク、調査タスク、実装計画、AI 実行、Draft PR 作成、レビュー、Eval を管理する。

### PRD-00-USER-2: P0 利用シナリオ

P0 で成立させる主要ユースケースは次のとおりである。

| ID | ユースケース | 説明 |
|---|---|---|
| PRD-00-UC-1 | チケット管理 | 個人プロジェクトの実装タスクを登録し、受け入れ条件、優先度、リスク、状態を管理する。 |
| PRD-00-UC-2 | Research 管理 | 調査テーマ、仮説、Claim、EvidenceSource、EvidenceItem、未解決論点、判断を管理する。 |
| PRD-00-UC-3 | Research-to-Ticket | 調査結果から実装可能なチケット、Acceptance Criteria、Risk Flags、Test Plan、Docs Update、Rollback Plan を生成する。 |
| PRD-00-UC-4 | 実装計画承認 | AI が作った分解案や実装計画を、人間が確認してから採用する。 |
| PRD-00-UC-5 | AI run 管理 | AgentRun の状態、イベント、コスト、失敗理由、ContextSnapshot、Artifact を追跡する。 |
| PRD-00-UC-6 | Draft PR flow | AI の実装結果を Draft PR まで進め、merge / deploy は P0 では実行しない。 |
| PRD-00-UC-7 | 監査と再現 | 誰が、どの AI に、どの文脈を渡し、何を承認し、どの結果になったかを追跡する。 |
| PRD-00-UC-8 | P0 Exit 評価 | private gold task 30-50 件で Hard Gates と Quality KPIs を測定する。 |

### PRD-00-USER-3: 将来の想定ユーザー

P1 以降では、チーム開発者、承認者、管理者、セキュリティ担当、リサーチ担当、商用利用の workspace owner を想定する。

ただし、P0 ではこれらの利用者を実装対象にしない。P0 では `tenant_id`、workspace、membership、actor / principal、policy、secret_ref、adapter などの境界を準備し、将来の multi-tenant 認証、external IdP、billing、support access、チーム承認へ移行できる構造を残す。

## 3. 解決する課題

### PRD-00-PROBLEM-1: Deep Research と実装が分断される

通常の Deep Research はレポート生成で終わりやすく、実装可能なチケット、受け入れ条件、テスト計画、rollback 方針、PR まで連続しない。

TaskManagedAI は ResearchTask、Claim、EvidenceSource、EvidenceItem、Decision、generated tickets を first-class に扱い、Research の結論を実装タスクへ接続する。

### PRD-00-PROBLEM-2: 証拠と判断の追跡が弱い

AI が出した結論だけを保存しても、後から根拠、引用、鮮度、矛盾、採用理由を検証できない。

TaskManagedAI は Claim / Evidence / Source / provenance_json を正規化し、どの証拠からどの判断やタスクが生まれたかを追跡する。

### PRD-00-PROBLEM-3: AI 実行がブラックボックス化する

AI が失敗した時に、どの prompt、model、provider、repo state、tool manifest、policy version、evidence set、承認履歴で実行したかが残らないと、改善も監査もできない。

TaskManagedAI は AgentRun、AgentRunEvent、ContextSnapshot、Artifact、Budget、Approval を保持し、実行状態ではなく実行履歴を正本にする。

### PRD-00-PROBLEM-4: AI 出力を安全に実行できない

AI 出力を直接 command、SQL、workflow、外部 tool 操作へ接続すると、prompt injection、secret leakage、excessive agency、dangerous command、forbidden path 書き込み、コスト暴走のリスクが高い。

TaskManagedAI は Output Validator、Input Trust Layer、Policy Engine、Approval Gate、Tool Registry、Runner Sandbox、BudgetGuard を組み合わせ、AI 出力と実行の間に検証と承認を挟む。

### PRD-00-PROBLEM-5: 承認と責任境界が曖昧になる

AI がどこまで自動実行してよいか、誰が承認したか、承認後に差分が変わっていないかが曖昧だと、実運用で使えない。

TaskManagedAI は action class、policy decision、approval request、audit event を管理し、read は自動、write は policy と承認、merge / deploy / secret access は P0 では deny または明示承認という境界を置く。

### PRD-00-PROBLEM-6: コストと効果が測れない

AI タスク管理ツールは、見た目の自動化ではなく、受け入れ条件達成率、time to merge、approval wait、citation coverage、cost per completed task を測定できなければ改善できない。

TaskManagedAI は Eval Harness と private gold task により、AI が本当に役に立っているかを定量評価する。

### PRD-00-PROBLEM-7: 個人運用と商用化準備の両立が難しい

最初から multi-tenant SaaS を作ると重く、個人専用の作り捨てにすると商用化時に作り直しになる。

TaskManagedAI は P0 を個人専用に絞りつつ、tenant 境界 invariant、secret_ref、adapter、policy、artifact、schema version を初期から入れるハイブリッド方針を採る。

## 4. 成功指標

### PRD-00-SUCCESS-1: P0 完了判定

P0 完了は次の式で判定する。

```text
P0 完了 = (Hard Gates 全件達成) AND (Quality KPI の未達数 <= 1)
```

Hard Gates は 1 件でも未達なら P0 承認不可とする。Quality KPIs は未達 1 個以下なら承認可、2 個以上なら改善 Sprint を追加する。

### PRD-00-SUCCESS-2: Hard Gates

| ID | Gate | 達成条件 |
|---|---|---|
| PRD-00-HARD-1 | `policy_block_recall` | 既知危険 fixture を 100% block する。 |
| PRD-00-HARD-2 | `secret_canary_no_leak` | fake API key の漏えい検知 0 件、AI 経由の外部送信 0 件。 |
| PRD-00-HARD-3 | `tenant_isolation_negative_pass` | 越境 SELECT / INSERT / UPDATE / DELETE が全試行で失敗する。 |
| PRD-00-HARD-4 | `backup_restore_rpo_rto` | RPO <= 24h、RTO <= 4h、PITR 検証成功。 |
| PRD-00-HARD-5 | `forbidden_path_block` | `.env`、`.git/config`、secrets、migrations 等への AI / runner 書き込みが全試行で失敗する。 |
| PRD-00-HARD-6 | `dangerous_command_block` | `rm -rf /`、`curl \| sh`、`chmod 777`、fork bomb 等を全試行で拒否する。 |
| PRD-00-HARD-7 | `prompt_injection_resist` | system 指示上書きや untrusted_content からの権限昇格試行が全件失敗する。 |

### PRD-00-SUCCESS-3: Quality KPIs

| ID | KPI | 初期閾値 |
|---|---|---|
| PRD-00-KPI-1 | `acceptance_pass_rate` | 受入条件達成率 >= 0.6 |
| PRD-00-KPI-2 | `time_to_merge` | median <= 2.0h、個人運用前提、mock merge |
| PRD-00-KPI-3 | `approval_wait_ms` | median <= 4h、個人運用前提、UI 通知前提 |
| PRD-00-KPI-4 | `citation_coverage` | claim with citation / total claim >= 0.9 |
| PRD-00-KPI-5 | `cost_per_completed_task` | <= $0.5/task、mini モデル中心 |

### PRD-00-SUCCESS-4: 定性指標

| ID | 指標 | 判断基準 |
|---|---|---|
| PRD-00-QUAL-1 | 作業の連続性 | Research、Ticket、Approval、AgentRun、Draft PR、Eval、Audit が 1 つの流れとして追える。 |
| PRD-00-QUAL-2 | 説明可能性 | AI がなぜその計画や判断を出したかを Evidence、ContextSnapshot、Artifact から説明できる。 |
| PRD-00-QUAL-3 | 安全境界 | AI に直接 secret、repo token、production deploy、merge 権限を渡さない。 |
| PRD-00-QUAL-4 | 再現性 | repo state、prompt pack、policy pack、tool manifest、provider request fingerprint、evidence set hash を保存できる。 |
| PRD-00-QUAL-5 | 運用可能性 | 個人運用で target effort 内に P0 を完走でき、over budget 時は defer line に従って P0.1 / P1 へ送れる。 |

## 6. 前提条件と制約

### PRD-00-CONSTRAINT-1: 技術前提

P0 の現実的な基準線は、単一 VPS、Docker Compose、PostgreSQL、Redis、FastAPI、Next.js、Python worker、Tailscale Serve/SSH、GitHub App、AI Agent Adapter である。

技術選択の詳細な API 形状、DB schema、service 分割、画面設計は基本設計で扱う。本書では、プロダクト要求上の前提と制約としてのみ扱う。

### PRD-00-CONSTRAINT-2: ネットワーク制約

P0 は Tailscale 閉域で運用し、Funnel は使わない。backend は localhost / internal network を前提とし、public internet へ直接公開しない。

Tailscale HTTPS の machine 名は Certificate Transparency 公開ログに載りうるため、machine naming は中立的な名前にする。

### PRD-00-CONSTRAINT-3: 商用化準備のハイブリッド方針

P0 は個人専用に絞るが、商用化の可能性を前提に、tenant 境界、workspace、membership、actor / principal、secret_ref、Provider Compliance Matrix、policy / audit、adapter 境界を初期から維持する。

Tailscale Personal は non-commercial 専用であるため、商用化検討時は Tailscale plan、tailnet 分離、external auth、billing、support access、Vault、RLS 有効化を再評価する gate を持つ。

### PRD-00-CONSTRAINT-4: ZDR / Provider Compliance

機密コードを扱う場合は、ZDR 対象機能のみ使用する方針を採る。Provider への送信可否は Provider Compliance Matrix で管理し、送信 payload の `payload_data_class` が provider の `allowed_data_class` 上限を超えるデータ送信は policy decision で deny する。

Provider の retention、training use、region / data transfer、subprocessor、plan_required、last_verified_at は外部仕様変更の影響を受けるため、Matrix に記録して定期確認する。

### PRD-00-CONSTRAINT-5: 秘密情報制約

DB には secret 値を保存せず、`secret_ref` のみ保存する。runner や AI に secret 値や repo token を直接渡さない。

P0 の標準は SOPS + age + API SecretBroker とし、将来 Vault へ切り替えられる境界を持つ。

### PRD-00-CONSTRAINT-6: 評価データ制約

P0 Exit は private gold task 30-50 件で計測する。Gold Task Seed v0 は TaskManagedAI 自体を dogfooding して作る。

secret、token、個人情報を含む task は sanitize または除外し、各 task の `payload_data_class=internal` 以下に制限する。

## 7. 重要な設計原則

### PRD-00-PRINCIPLE-1: AI 出力直結禁止

AI 出力を直接 command、SQL、workflow、外部 tool 操作へ接続しない。必ず artifact 化し、schema validation、policy lint、approval、runner sandbox を通す。

### PRD-00-PRINCIPLE-2: Deny-by-default

ネットワーク、tool、repo 操作、secret access、merge、deploy は deny-by-default とする。許可は action class、allowlist、policy、承認により明示する。

### PRD-00-PRINCIPLE-3: 最小権限

GitHub App、SecretBroker、runner、Provider Adapter、Tool Gateway は、実行に必要な最小権限だけを持つ。短命 token と capability token を優先する。

### PRD-00-PRINCIPLE-4: 人間承認を workflow に埋め込む

AI の分解案、実装計画、repo write、secret access、high-risk change は人間承認を経る。merge / deploy は P0 では対象外または常時 deny とする。

### PRD-00-PRINCIPLE-5: Evidence first-class

Deep Research の出力は要約文だけで保存しない。Claim、EvidenceSource、EvidenceItem、provenance_json、citation coverage を first-class にする。

### PRD-00-PRINCIPLE-6: 実行履歴を正本にする

AgentRun の現在状態だけでなく、append-only AgentRunEvent、Artifact、ContextSnapshot、Approval、PolicyDecision、AuditEvent を正本にする。

### PRD-00-PRINCIPLE-7: 再現性を要件に含める

repo state、prompt pack、policy pack、tool manifest、evidence set hash、provider request fingerprint、provider continuation reference を保存し、後から実行条件を説明できるようにする。

### PRD-00-PRINCIPLE-8: 評価可能性を P0 Exit に組み込む

AI が役に立ったかを主観で判断しない。Hard Gates と Quality KPIs を P0 Exit Criteria として固定し、未達時は改善 Sprint を追加する。

### PRD-00-PRINCIPLE-9: Over budget 時に defer する

個人運用で完走可能にするため、各 Sprint は target_days / max_days / defer_if_over_budget を持つ。max を超えそうな場合は P0.1 または P1 へ送る。

### PRD-00-PRINCIPLE-10: Sprint Pack を実装前 gate にする

各機能単位 Sprint は Sprint Pack を作り、目的、対象外、受け入れ条件、検証手順、残リスクを明確にしてから実装する。高リスク領域は ADR を必須にする。

## 9. P0.1+ Multi-Agent Vision (2026-05-10 追記、Phase A-E 経由 56 finding 全件 adopt 後の正式 vision)

P0 完了後 (Sprint 13 以降) に、TaskManagedAI を **「AI 集合体 = 一つの会社」** メタファーで稼働させる multi-agent vision を導入する。本 section は P0 以後のロードマップとして PRD-00 に追加.

### 9.1 5 core 要件

1. **AI Society = 会社メタファー**: 司令塔 (orchestrator) + 専門 agent 群 (implementer / reviewer / tester / security_agent / researcher / observer / curator / dispatcher / repair_specialist) + 完全自律運営
2. **Multi-agent orchestration + 完全自律**: 1 ticket = N AgentRun 並列、司令塔が自律 dispatch、inter-agent communication (レビュー / 話し合う場)、Discord 等外部 trigger 不要
3. **UI ↔ CLI parity**: TaskManagedAI Web UI でできること = CLI (`tm`) でできること、結果は同 DB で同一
4. **メモリー記録 (UI/CLI 両方)**: ユーザー手動記録 + 司令塔自動記録、user-scope ハーネス v5 (hermes 級 memory) を backend として活用
5. **キャラクター / Graphic (P2)**: AI agent をキャラクター化 (Codex 画像生成 or default emoji icon)

### 9.2 既存 invariant 不変保証

P0 完了時の以下 invariant は P0.1+ で **すべて不変**:

- AgentRun 16 状態 + blocked サブ 3 (event_type 22→31 は additive 拡張のみ)
- ContextSnapshot 10 列
- Provider Compliance v2 13 reason_code
- SecretBroker atomic claim + OperationContext fingerprint
- Approval Workflow 4 整合 + decider human-only + self-approval 禁止
- Cross-source enum 5+ source 整合
- 既存 action_class 7 種 (新 operation は event_type / Tool Registry / policy_profile に分散)
- 全 id=uuid + tenant_id=bigint
- 3 gateway 分離 (tool_mutating_gateway_stub / runner_mutation_gateway / remote_agent_gateway)

### 9.3 Phase 構成 (P0 後)
