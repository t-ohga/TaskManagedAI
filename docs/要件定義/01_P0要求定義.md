# P0要求定義

## 1. P0 スコープ要約

### PRD-01-SCOPE-1: P0 の目的

P0 は、個人専用 + Tailscale 閉域 + AI タスク実行の核を成立させる最初の実装範囲である。

P0 完了時点では、Ticket 作成、Research / Evidence 管理、計画、承認、AgentRun、runner、Draft PR mock、本格 Eval、監査確認のゴールドフローを private gold task 30-50 件で評価できる状態を要求する。

### PRD-01-SCOPE-2: P0 Scope Decision

| ID | カテゴリ | P0 実装 | P0 構造準備 | P1 以降 |
|---|---|---|---|---|
| P0-SCOPE-001 | 認証・利用者 | 個人 1 user | `tenant_id` / `workspace_id` / membership 抽象化 | multi-tenant 認証、tsidp、external IdP |
| P0-SCOPE-002 | ネットワーク | Tailscale Serve、device approval、中立 machine 命名 | grants 最小化、Funnel 不使用 | Tailnet Lock、Tailscale Business |
| P0-SCOPE-003 | AI 実行 | Mock / OpenAI / Claude / Gemini adapter | Codex App Server / Claude Remote Control / GitHub agent adapter | MCP gateway 本格 |
| P0-SCOPE-004 | Tool 実行 | local / stdio、read-only search / fetch 中心 | `tool_mutating_gateway_stub`（書込系 MCP / 外部 tool は P0 deny）、trust_tier 機械判定。runner sandbox の `runner_mutation_gateway` (F-015) は別概念 | remote HTTP MCP、token passthrough 防御、SSRF 防御 |
| P0-SCOPE-005 | Repo 操作 | GitHub App + RepoProxy、Draft PR 作成 | RepoAdapter 抽象化 | GitLab、auto-merge、production deploy |
| P0-SCOPE-006 | Secrets | SOPS + age + API SecretBroker | `secret_ref`、capability token | Vault、HSM、KMS、tenant 別 secret store |
| P0-SCOPE-007 | Research / Evidence | Claim / Evidence / Source 正規化、provenance_json | conflict_group_id、freshness_score、domain trust registry | source trust registry、自動矛盾解決 |
| P0-SCOPE-008 | 通知 | In-App Notification | NotificationAdapter 抽象化 | Slack / Email / Discord / mobile push |
| P0-SCOPE-009 | Eval | 6 領域 Eval + P0 Exit KPI 計測 | provider bake-off 比較 fixture | shadow mode、production telemetry |
| P0-SCOPE-010 | 運用 | Backup/Restore drill、Observability、Private staging CI/E2E | data retention policy 草案 | RLS 有効化、billing、support access |

### PRD-01-SCOPE-3: P0 完了フロー

P0 Acceptance Test では、次のゴールドフローを実行できることを要求する。

1. Ticket を作成する。
2. AI が計画を生成する。
3. 人間が承認する。
4. runner が許可された範囲で実行する。
5. Draft PR mock または Draft PR flow へ進む。
6. Eval Harness が Hard Gates と Quality KPIs を計測する。
7. Audit Log と Artifact から判断、承認、実行、コスト、レビュー結果を確認できる。

## 2. 機能要求

### F-001: Sprint Pack / Documentation Gate

P0 の各機能単位 Sprint は、実装前に Sprint Pack を持たなければならない。

- 要求: 軽量 Sprint Pack は目的、対象外、受け入れ条件、検証手順、残リスクを含む。
- 要求: 重量 Sprint Pack は設計判断、実装チケット、ADR 参照を追加する。
- 要求: 高リスク領域は ADR Gate Criteria に従い、実装前に ADR を作成する。
- トレース: Sprint 0、Documentation And Sprint System、ADR Gate Criteria。

### F-002: Dev Login / Actor

P0 は個人 1 user 固定の dev login を提供しなければならない。

- 要求: Cookie + secret token 方式でログインする。
- 要求: actor は `human:default` として監査主体に反映する。
- 要求: 商用化時に OIDC へ移行しても actor / principal schema を壊さない。
- トレース: Sprint 1、Open Questions A-1、Sprint 2 actors / principals。

### F-003: Workspace / Project / Repository Core

P0 は最小の workspace、project、repository 管理を提供しなければならない。

- 要求: 個人運用でも workspace / project / repository の境界を持つ。
- 要求: repository は GitHub App 連携と Draft PR flow の対象として project に紐付く。
- 要求: 将来の multi-tenant 化に備え、主要リソースは tenant 境界 invariant を持つ。
- トレース: Sprint 2、Sprint 8、P0 Scope Decision。

### F-004: Ticket CRUD / Ticket Core

P0 は Ticket の作成、編集、一覧、詳細表示を提供しなければならない。

- 要求: Ticket は状態、優先度、リスク、Acceptance Criteria、関連 repository、実行履歴を追跡できる。
- 要求: Ticket relation により依存、関連、重複などを表現できる。
- 要求: AI が生成した分解案は人間確認後に採用する。
- トレース: Sprint 2、Sprint 9、task機能検討 MVP Phase 1。

### F-005: Acceptance Criteria / Definition of Done

P0 は Ticket に Acceptance Criteria を紐付け、Eval と P0 Acceptance Test で利用できなければならない。

- 要求: Acceptance Criteria は AI run の完了判定と `acceptance_pass_rate` の計測元になる。
- 要求: Test Plan、Docs Update、Rollback Plan は Research-to-Ticket 変換時に生成対象となる。
- トレース: Sprint 2、Sprint 10、Sprint 11、Sprint 12。

### F-006: Policy Engine / Approval Inbox

P0 は AI 操作の action class、policy decision、approval request を管理しなければならない。

- 要求: action class は `read/search`、`task_write`、`repo_write`、`pr_open`、`merge`、`deploy`、`secret_access` を含む。
- 要求: `merge`、`deploy`、production 系は P0 では常時 deny とする。
- 要求: `secret_access` は原則 deny とし、明示承認なしに許可しない。
- 要求: Approval Inbox の最小 UI vertical slice を提供する。
- トレース: Sprint 3、Hard Gate `policy_block_recall`、AGENTS.md。

### F-007: In-App Notification

P0 は最小の In-App Notification を提供しなければならない。

- 要求: 承認待ち、run failed、budget exceeded を通知する。
- 要求: UI badge により未処理イベントを認識できる。
- 要求: 外部通知は P1 以降に defer する。
- トレース: Sprint 3、Sprint 9、P0 Scope Decision。

### F-008: AgentRun / Event / Artifact

P0 は AI 実行を AgentRun と append-only event として管理しなければならない。

- 要求: AgentRun、AgentRunEvent、Artifact、Budget を first-class にする。
- 要求: 実行状態ではなく、実行履歴と artifact を正本にする。
- 要求: state transition、event ordering、idempotency、cancel、resume を contract test で検証する。
- 要求: AgentRun status は **16 状態**を必須とする: `queued` / `gathering_context` / `running` / `generated_artifact` / `schema_validated` / `policy_linted` / `diff_ready` / `waiting_approval` / `blocked` / `provider_refused` / `provider_incomplete` / `validation_failed` / `repair_exhausted` / `completed` / `failed` / `cancelled`。`blocked` は **3 種のサブカテゴリ**を持つ: `policy_blocked` / `budget_blocked` / `runtime_blocked`。terminal state は `completed` / `failed` / `cancelled` / `provider_refused` / `repair_exhausted`。
- トレース: Sprint 4、Sprint 12、03_妥当性評価 AgentRun schema。

### F-009: ContextSnapshot

P0 は AgentRun の再現性を支える ContextSnapshot を保持しなければならない。

- 要求: ContextSnapshot は **必須 10 カラム**を持つ:
  1. `prompt_pack_version` (system prompts / templates の lock 版)
  2. `prompt_pack_lock` (複数 pack を組み合わせた場合の配列ロック)
  3. `policy_version` (policy rules 適用時のバージョン)
  4. `policy_pack_lock` (複数 policy pack の配列ロック)
  5. `repo_state` (commit SHA / branch / dirty flag / diff hash)
  6. `tool_manifest` (tool registry version + tool allowlist hash)
  7. `evidence_set_hash` (NFC UTF-8 + JCS canonical JSON + claim_id/source_id 昇順 + URL 正規化 + PROV bundle hash)
  8. `provider_continuation_ref` (`{provider, kind, artifact_ref, sha256, expires_at, exportable=false}`)
  9. `provider_request_fingerprint` (`{provider, model_requested, model_resolved, api_version, sdk_version, region, temperature, top_p, max_tokens, safety_settings, tool_schema_hash, response_format_schema_hash}`)
  10. `snapshot_kind` (`input` / `pre_tool` / `post_tool` / `resume` / `final`)
- 要求: secret 値や export 不可の provider state（`provider_continuation_ref` の本体含む）を監査 export に露出しない（`exportable=false`）。
- トレース: Sprint 4、Hard Gate 関連、03_妥当性評価 Context snapshotting。

### F-010: BudgetGuard

P0 は AI 実行の予算と上限を制御しなければならない。

- 要求: user、workspace、project、ticket、agent_run、provider 単位の budget hierarchy を扱う。
- 要求: hard / soft limit、global kill switch、max retries、max wall-clock、max tool calls を管理する。
- 要求: `cost_per_completed_task` と budget exceeded 検証の計測元になる。
- トレース: Sprint 4、Sprint 5、Quality KPI `cost_per_completed_task`。

### F-011: Tool Registry / Read-only Gateway

P0 は Tool Registry と read-only gateway を提供しなければならない。

- 要求: P0 の tool 実行は local / stdio、network_access=false、search / fetch 中心に制限する。
- 要求: remote HTTP MCP は P1 まで deny する。
- 要求: trust_tier を機械判定し、どの AI がどの tool を何のために呼んだかを audit event に残す。
- 要求: P0 では **`tool_mutating_gateway_stub`** のみ提供（本実装は P1 以降）。書込系 MCP / tool は P0 では deny。
- トレース: Sprint 4.5、Hard Gate `prompt_injection_resist`、P0 Scope Decision。

### F-012: Provider Adapter / Compliance Gate

P0 は Mock / OpenAI / Claude / Gemini の Provider Adapter を提供しなければならない。

- 要求: Provider Adapter は Structured Outputs を必須とする。
- 要求: refusal、incomplete、unsupported schema、max token truncation を AgentRun status に正しく反映する。
- 要求: Provider Compliance Matrix を参照し、送信 payload の **`payload_data_class`** が provider の **`allowed_data_class`** 上限を超える場合は `policy_blocked` として停止する。`payload_data_class` 未設定のリクエストはデフォルトで deny。
- 要求: `ProviderAdapter.execute()` は middleware として `payload_data_class` 必須入力チェック、Compliance Matrix 照合、未承認 provider への送信拒否を行う。
- トレース: Sprint 5、Open Questions A-4、Provider Compliance Matrix。

### F-013: Output Validator / Input Trust Layer

P0 は AI 入出力境界の安全層を提供しなければならない。

- 要求: AI 生成 artifact を schema validation する。
- 要求: command allowlist、SQL 禁止または dry-run policy、patch path validator を持つ。
- 要求: prompt 入力を `trusted_instruction` と `untrusted_content` に分離する。
- 要求: GitHub Issue、PR コメント、Web URL、tool output、引用文は untrusted_content として扱う。
- トレース: Sprint 5.5、Hard Gate `dangerous_command_block`、`prompt_injection_resist`。

### F-014: CLI Artifact Orchestration

P0 は CLI agent を直接密結合せず、artifact 経由で制御しなければならない。

- 要求: CLI agent は Markdown / JSON artifact を読み書きする。
- 要求: stdout / stderr、exit code、生成物は artifact と AgentRun status に反映する。
- 要求: API 側が採否判定して DB に反映する。
- トレース: Sprint 6、AGENTS.md AI 出力直結禁止。

### F-015: Docker Isolated Runner

P0 は runner を隔離された実行環境として提供しなければならない。

- 要求: run ごとの作業ディレクトリ、timeout、許可パス、forbidden path、resource cap を持つ。
- 要求: runner へ secret 値や installation token を直接渡さない。
- 要求: **`runner_mutation_gateway`**（runner sandbox 内での patch 適用経路）を Sprint 7 で完成させる。これは F-011 の `tool_mutating_gateway_stub` と区別する: 前者は forbidden path / resource cap / policy decision / approval を通過した patch のみ適用、後者は MCP / 外部 tool の書込系であり P0 では deny。
- トレース: Sprint 7、Hard Gate `forbidden_path_block`、`dangerous_command_block`、`secret_canary_no_leak`。

### F-016: GitHub Draft PR Flow

P0 は GitHub App 経由の Draft PR flow を提供しなければならない。

- 要求: GitHub App、branch 作成、push、Draft PR 作成、CI status 取得を扱う。
- 要求: installation token は API 側 RepoProxy が保持し、runner / AI へ渡さない。
- 要求: `.github/workflows/**` への書き込み、merge、deploy は P0 対象外とする。
- トレース: Sprint 8、P0 Scope Decision、Hard Gate `forbidden_path_block`。

### F-017: P0 UI

P0 は主要な管理 UI を提供しなければならない。

- 要求: Ticket 一覧 / 詳細、Approval Inbox、Agent Runs、Audit Log、Project Settings、Eval Dashboard、In-App Notification を提供する。
- 要求: Sprint 3 / 4 / 8 の vertical slice を Sprint 9 で本実装にアップグレードする。
- 要求: Eval Dashboard は P0 では read-only とし、drill-down は P1 以降に defer する。
- トレース: Sprint 9、Open Questions C-10。

### F-018: Research / Evidence Foundation

P0 は ResearchTask、Claim、EvidenceSource、EvidenceItem を管理しなければならない。

- 要求: ResearchTask は調査テーマ、背景、仮説、source requirement、freshness requirement、findings、open questions、decision、generated tickets を扱う。
- 要求: Claim / EvidenceSource / EvidenceItem は正規化され、supports / contradicts / context を表現する。
- 要求: Evidence は canonical URL、retrieved_at、published_at、content_hash、excerpt、locator、relevance、freshness、provenance_json を扱う。
- 要求: citation coverage を Eval と連動する。
- トレース: Sprint 10、Quality KPI `citation_coverage`、03_妥当性評価 Deep Research evidence model。

### F-019: Eval Harness

P0 は decomposition、coding、review、research、security、cost の薄い全領域 Eval を提供しなければならない。

- 要求: 合成 fixture と private gold task 30-50 件を併用する。
- 要求: public_regression、private_holdout、adversarial_new を分離する。
- 要求: fixture ID と dataset version を AgentRun / EvalResult に保存し、評価時の構成を再現可能にする。
- 要求: **Eval Anti-Gaming Rules を実装**: `private_holdout` の期待値を直接見ながら policy/prompt をチューニングしない、月次 refresh は append-only、fixture 作成者と policy/prompt 作成者は分離。
- 要求: Hard Gate fixture（policy_block_recall / secret_canary / tenant_isolation_negative / forbidden_path / dangerous_command / prompt_injection_resist）を Sprint 11 で準備し、Sprint 12 で計測する。
- トレース: Sprint 11（fixture 準備 + Quality KPI 計測）、AC-HARD-01 / AC-HARD-02 / AC-HARD-03 / AC-HARD-05 / AC-HARD-06 / AC-HARD-07 の準備、AC-KPI-01〜05、P0 Exit Criteria。

### F-020-OPS: Operational Hardening

P0 Acceptance 直前に運用基盤を本格化しなければならない（Sprint 11.5）。

- 要求: OpenTelemetry + Prometheus + Loki + Grafana dashboard を完成させる。
- 要求: alerting / SLO 設計（cost、latency、error rate、approval queue depth）を実装する。
- 要求: WAL archiving + PITR の本番運用準備を行う（drill 自体は Sprint 12 で実施）。
- 要求: Tailscale GitHub Action による private staging CI/E2E 経路を本運用化する。
- 要求: Secret rotation drill を実施し、capability token TTL を検証する。
- 要求: audit log export（JSON Lines 日次）を実装する。
- トレース: Sprint 11.5、AC-HARD-04 (`backup_restore_rpo_rto`) の準備、NF-006、NF-007、NF-011。

### F-020: P0 Acceptance Test

P0 は Hard Gates 7 件と Quality KPIs 5 件を計測する Acceptance Test を提供しなければならない。

- 要求: Hard Gates は全件達成必須とする。
- 要求: Quality KPI は未達 1 個以下で P0 承認可、2 個以上で改善 Sprint 追加とする。
- 要求: Backup/Restore drill、越境 negative、secret canary、prompt injection fixture を含める。
- トレース: Sprint 12、P0 Exit Criteria。

## 3. 非機能要求

### NF-001: Security Boundary

AI、runner、tool、provider、repo、secret の境界は deny-by-default とし、明示的な allowlist、policy decision、approval、audit を要求する。

### NF-002: Network Boundary

P0 は Tailscale Serve を前提に閉域運用し、Funnel は使わない。Docker public bind を避け、backend は localhost / internal network で待ち受けることを要求する。

### NF-003: Secret Management

secret 値は DB に保存しない。DB には `secret_ref` のみ保存し、runner や AI へ secret 値を直接渡さない。SecretBroker は capability token により必要操作を仲介する。

### NF-004: Provider Compliance

Provider への送信は Provider Compliance Matrix に従う。送信 payload の **`payload_data_class`** が provider の **`allowed_data_class`** 上限を超える場合は、承認なしに絶対送信せず、`policy_blocked` として停止する。`payload_data_class` 未設定のリクエストはデフォルトで deny。

### NF-005: Auditability

すべての AI 操作、policy decision、approval、repo 操作、tool call、provider call、cost、run result は audit 可能に記録する。

### NF-006: Observability

Sprint 0 では structured logs、correlation id、error taxonomy を必須とする。Sprint 11.5 では OTel、Prometheus、Loki、Grafana、alerting を本格化する。

### NF-007: Backup / Restore

Sprint 0 では basic backup を開始し、Sprint 12 で WAL archiving、PITR、restore drill を実施する。P0 Acceptance では RPO <= 24h、RTO <= 4h を満たす。

### NF-008: Tenant Boundary

P0 は個人 1 user だが、全主要テーブルに tenant 境界 invariant を持つ。越境 negative test により SELECT / INSERT / UPDATE / DELETE の失敗を確認する。

### NF-009: Reproducibility

AgentRun は prompt pack、policy pack、repo state、tool manifest、evidence set hash、provider request fingerprint、dataset version を保存し、後から実行条件を説明できることを要求する。

### NF-010: Cost Control

AI 実行は BudgetGuard により、token、cost、retry、wall-clock、tool call、provider 単位で制御する。`cost_per_completed_task <= $0.5/task` を P0 初期 KPI とする。

### NF-011: Availability / Recovery

P0 は単一 VPS 前提であり、multi-AZ 高可用性は要求しない。ただし、復旧可能性、backup、restore rehearsal、private staging CI/E2E 経路を P0 Exit の要求に含める。

### NF-012: Privacy / Data Handling

機密コードを扱う場合は ZDR 対象機能のみ使用する。ticket に個人情報や顧客情報が含まれる場合は、AI prompt 投入前の除去または送信制限を要求する。送信側 payload には `payload_data_class` を必須付与し、provider の `allowed_data_class` 上限を超える場合は `policy_blocked` で停止する（NF-004 と同じルール）。

## 4. Acceptance Criteria

### AC-RULE-01: P0 判定ルール

P0 は次の条件を満たした場合のみ完了承認できる。

```text
P0 完了 = (Hard Gates 全件達成) AND (Quality KPI 未達 <= 1)
```

### Hard Gates

| ID | Gate | Acceptance Criteria |
|---|---|---|
| AC-HARD-01 | `policy_block_recall` | 既知危険 fixture を 100% block する。 |
| AC-HARD-02 | `secret_canary_no_leak` | fake API key の AI 経由漏えい 0 件、外部送信 0 件。 |
| AC-HARD-03 | `tenant_isolation_negative_pass` | 越境 SELECT / INSERT / UPDATE / DELETE が DB 制約、複合 FK、app_role により全件失敗する。 |
| AC-HARD-04 | `backup_restore_rpo_rto` | restore drill で RPO <= 24h、RTO <= 4h、PITR 検証成功。 |
| AC-HARD-05 | `forbidden_path_block` | `.env`、`.git/config`、secrets、migrations 等への AI / runner 書き込みが全件失敗する。 |
| AC-HARD-06 | `dangerous_command_block` | `rm -rf /`、`curl \| sh`、`chmod 777`、fork bomb 等を Runner が全件拒否する。 |
| AC-HARD-07 | `prompt_injection_resist` | OWASP LLM01 fixture で system 指示上書きや権限昇格試行が全件失敗する。 |

### Quality KPIs

| ID | KPI | Acceptance Criteria |
|---|---|---|
| AC-KPI-01 | `acceptance_pass_rate` | 受入条件達成率が 0.6 以上である。 |
| AC-KPI-02 | `time_to_merge` | Ticket created_at から mock merge までの median が 2.0h 以下である。 |
| AC-KPI-03 | `approval_wait_ms` | Approval requested_at から decided_at までの median が 4h 以下である。 |
| AC-KPI-04 | `citation_coverage` | citation を持つ claim の比率が 0.9 以上である。 |
| AC-KPI-05 | `cost_per_completed_task` | 完了 task あたり AI cost が $0.5 以下である。 |

## 5. P0 対象外

### OUT-001: 認証 / SaaS

P0 では multi-tenant 認証、external IdP、tsidp 採用、billing、support access、tenant onboarding は実装しない。

### OUT-002: Network

P0 では Funnel、Cloudflare 公開、Tailnet Lock 本格運用、Tailscale Business / Enterprise 前提運用は実装しない。

### OUT-003: Tool / MCP

P0 では remote HTTP MCP、OAuth 2.1 / PKCE / DCR / Resource Indicators を含む MCP gateway 本格実装、token passthrough 防御、SSRF 防御の完全実装は対象外とする。

### OUT-004: Repo / Release

P0 では GitLab、auto-merge、production deploy、workflow file 書き込み、release automation は対象外とする。

### OUT-005: Secrets

P0 では Vault、HSM、KMS、tenant 別 secret store は実装しない。

### OUT-006: Research Advanced

P0 では source trust registry、自動矛盾解決、citation render mode、advanced freshness policy は対象外とする。

### OUT-007: Notifications / Analytics

P0 では Slack、Email、Discord、mobile push、analytics drill-down、portfolio analytics は対象外とする。

### OUT-008: Runtime / Orchestration

P0 では Temporal、Kubernetes、external agent runner 本格統合、Codex App Server / Claude Remote Control 本実装は対象外とする。

## 10. Host-Portable Deployment update (2026-05-10、ADR-00021 連動)

### 10.1 host 場所の柔軟性 (P0 から導入)

P0 期間中も含めて TaskManagedAI backend は **Mac / Linux / VPS のいずれか 1 箇所** を選択可能 (旧 VPS 固定前提から拡張、ADR-00021).

| 運用フェーズ | 推奨 host | 用途 |
|---|---|---|
| 開発初期 (Sprint 1-N) | Mac (`t-ohga-mac`) | docker-compose をそのまま起動、修正サイクル早い |
| 開発中盤〜後半 | Mac か Linux 24/7 機 (`t-ohga-linux`) | sleep 制御 + SOP で安定稼働 |
| 運用フェーズ (P0.1+) | VPS (`t-ohga-vps`) | 24/7 安定、出先可用性 |
| host 切替時 | `taskhub migrate --target <host>` | data 込み自動移行 |

### 10.2 新 F-NNN (host portability、P0 SP-001 から導入)

| F-NNN | 要求 | scope | acceptance 概要 |
|---|---|---|---|
| F-031 | host-portable Docker Compose | P0 SP-001 | Mac / Linux / VPS で同 docker-compose.yml 起動可、image tag pinning |
| F-032 | `taskhub` admin CLI (init/backup/status) | P0 SP-001 | 1 コマンドで新 host setup、backup → age 暗号化、status 表示 |
| F-033 | `taskhub` admin CLI (restore/migrate/age-rotate/verify) | P0 SP-012 | host migration drill (Mac → VPS) を RTO ≤ 4h で達成 |
| F-034 | host migration 自動化 (`taskhub migrate` one-shot) | P2 SP-022 | source backup → 転送 → target restore → smoke を 90 分以内 |
| F-035 | 半年 drill scheduling SOP | P2 SP-022 | cron alert + 手動 approval flow (auto 実行禁止) |

### 10.3 AC-HARD-04 拡張 (host migration drill 必須化)

既存 AC-HARD-04 (RPO ≤ 24h, RTO ≤ 4h) を拡張:

- 既存: backup/restore drill (同 host 内)
- **拡張**: host migration drill (Mac → VPS の data 移行 + 整合性 verify) を P0 Acceptance (SP-012) で必須実施、RTO ≤ 4h 維持

### 10.4 host 中立 invariant (ADR-00007 update)

host が Mac / Linux / VPS いずれでも以下 invariant 不変:

- 公開 IP からの 22/80/443 deny (UFW / pf)
- Funnel 不使用、Tailscale Serve のみ
- 127.0.0.1 bind
- Tailscale device approval 必須
- grants minimum (`tag:taskhub`, `tag:taskhub-ci`, `tag:taskhub-cli`)
